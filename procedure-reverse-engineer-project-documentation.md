# Reverse Engineer Project Documentation Procedure

## Purpose

This procedure derives a complete documentation set from an existing codebase.

The procedure works from the smallest implementation units outward. It first documents modules, then groups them into subsystems, then derives architecture, then derives functional behavior, then creates a project README that points to the most important entry pages.

## Inputs

- Existing source code.
- Existing automated tests.
- Existing functional specifications, design documents, procedures, backlog files, wiki pages, and README files.
- Build, test, lint, and runtime commands when the project defines them.
- Current project conventions for documentation root, source folders, test folders, and wiki root.

## Output Set

- Module design documents.
- High-level design documents.
- Architecture document.
- Functional specification documents.
- Project README.
- Project wiki updates when the project uses docs/wiki.
- Open questions for behavior, ownership, or source conflicts that cannot be resolved from the repository.

## Source Authority

Use this authority order when sources disagree:

1. Code and tests describe actual behavior.
2. Functional specifications and requirements describe intended behavior.
3. Project procedures and agent instructions describe workflow obligations.
4. Backlog files describe tracked work and known status.
5. Architecture, high-level design, module design, and plan documents describe design intent.
6. Wiki pages summarize and navigate the sources above.

Do not invent behavior to fill documentation gaps. Record an open question when the repository does not provide enough evidence.

## Wiki-Compatible Document Contract

All generated functional and technical documents are wiki page subclasses.

Start each generated document with the shared wiki-compatible sections:

- Current Understanding.
- Authoritative Sources.
- Related Code.
- Related Tests.
- Related Backlog Items.
- Related Wiki Pages.
- Open Questions.
- Maintenance Notes.

After the shared sections, append the specialized sections from the relevant template. This allows module designs, high-level designs, architecture documents, and functional specifications to live under docs/wiki without losing the wiki page contract or forcing the base project wiki template to know every specialized page type.

When a target project keeps generated documents outside docs/wiki, keep the same inherited structure. The wiki can then link those documents as durable sources or move them into docs/wiki later without rewriting their format.

Diagram sections are optional subclass sections. Use Mermaid as the editable diagram source when a section defines multiple items whose meaning depends on sequence, association, aggregation, containment, ownership, dependency direction, lifecycle state, coverage, or data flow. Use rendered SVG only as a companion artifact for review or publishing surfaces that cannot render Mermaid reliably, and keep the editable source authoritative.

## Pass 0: Repository Orientation

Start with a short orientation pass before writing documentation.

1. Identify the source roots, test roots, application entry points, scripts, configuration files, migrations, public routes, command entry points, and generated artifacts.
2. Identify existing documentation roots and note whether they already contain functional, architecture, high-level design, module design, or wiki pages.
3. Identify build and test commands from project metadata and procedures.
4. Identify current worktree status when the project is a Git repository.
5. Record obvious documentation gaps and conflicts for later passes.

Completion gate:

- The documentation root is known.
- The source and test roots are known.
- Existing documentation is inventoried.
- The verification commands are known or recorded as not yet identified.

## Pass 1: Module Design Documents

Reverse engineer each module before grouping modules into larger structures.

1. Walk the source roots and identify modules by runtime responsibility, not only by file name.
2. For each module, read its implementation, callers, imports, exported contracts, tests, configuration use, persistence use, and user-visible behavior.
3. Create or update one module design document per meaningful module or tightly coupled module folder.
4. Use the module design template for shared wiki sections, runtime path, parent context, responsibilities, callers, dependencies, public contracts, internal state, processing rules, invariants, configuration, external interfaces, UI behavior, error handling, and verification.
5. Add a Processing Diagram only when conditional flow, retries, error handling, or state transitions are difficult to understand from prose alone.
6. Link related tests and source files from each module document.
7. Record missing tests, ambiguous ownership, and undocumented side effects as open questions.

Module boundary rules:

- Prefer one module document for one coherent responsibility.
- Split a file or folder into more than one documented module only when it contains clearly separate responsibilities that can change independently.
- Keep helper functions inside the owning module document unless they are reused independently across subsystem boundaries.
- Do not create a module document for generated artifacts unless the generated artifact is the maintained source of runtime behavior.

Completion gate:

- Every important source module has a module design document or an explicit deferral.
- Every module design document links its source and tests.
- Open questions capture unresolved ownership or behavior.

## Pass 2: High-Level Design Documents

Group documented modules into subsystems after the module pass is complete.

1. Review all module design documents and identify clusters of modules that collaborate to deliver one subsystem, feature family, workflow engine, integration, UI area, or operational capability.
2. Create or update one high-level design document per subsystem.
3. Use the high-level design template for shared wiki sections, parent architecture, scope, current data anchors, constituent components, interaction model, lifecycle, data shapes, configuration, implementation order, invariants, non-goals, definition of good, and verification.
4. Add high-level design diagrams for structural relationships inside the sections they clarify, including scope boundaries, data anchors, constituent component associations, interaction flow, lifecycle states, data contracts, configuration ownership, implementation sequence, and verification coverage when those relationships appear in the subsystem.
5. Link every constituent component document from the subsystem document, including module design documents when the component is a code module.
6. Link functional specs when a subsystem directly serves user-visible workflows.
7. Record overlapping ownership and missing subsystem boundaries as open questions.

Subsystem boundary rules:

- Prefer subsystems that can be explained by runtime collaboration and ownership.
- Do not group modules only because they live in the same folder.
- Do not split a subsystem only because it has several implementation layers.
- Keep cross-cutting concerns separate when they impose rules across multiple subsystems.

Completion gate:

- Every important module belongs to a subsystem or has an explicit reason for standing alone.
- Every subsystem document links its constituent modules.
- High-level documents describe real interactions instead of restating module summaries.

## Pass 3: Architecture Document

Derive architecture from the completed high-level and module design documents.

1. Review the full set of high-level design documents.
2. Read module documents only where subsystem boundaries, dependency direction, state ownership, technology choices, or cross-cutting rules need evidence.
3. Create or update the architecture document using the architecture template.
4. Document the shared wiki sections plus system purpose, scope, system context, technology stack, file organization, architectural layers, key components, data flow, lifecycle flow, cross-cutting concerns, design principles, invariants, risks, trade-offs, and verification.
5. Add architecture diagrams for structural relationships, including scope boundaries, system context, file ownership, layers, component associations, data movement, lifecycle states, concern ownership, and verification coverage when those relationships appear in the architecture.
6. Link all high-level design documents from the architecture document.
7. Record architecture-level contradictions between code, tests, and documents as open questions.

Architecture derivation rules:

- Architecture must describe the system that exists, not the system that would be ideal.
- Architecture may call out drift and risk, but it must not silently rewrite behavior.
- Architecture should explain dependency direction and ownership without duplicating every module detail.
- Architecture should mention technology choices only when they shape boundaries, runtime behavior, or project-wide rules.

Completion gate:

- The architecture document covers all major subsystems.
- Cross-cutting rules are explicit.
- Verification evidence is linked or recorded as missing.

## Pass 4: Functional Specification Documents

Derive functional documentation after technical structure is understood.

1. Review the architecture document, high-level design documents, module design documents, routes, UI components, command entry points, integration points, tests, and existing product documents.
2. Identify user-visible workflows, admin workflows, operator workflows, external-system workflows, route behavior, permissions, statuses, error states, and acceptance behavior.
3. Create or update functional specifications using the functional spec template, including the shared wiki sections before the functional extension sections.
4. Write workflows from the actor's point of view.
5. Add a Workflow Diagram when actor paths, branching states, permissions, or external handoffs are clearer visually.
6. Link the technical documents and modules that implement each workflow.
7. Link tests that verify each workflow, state, edge case, and permission rule.
8. Record mismatches between intended behavior and actual code behavior as open questions or known defects.

Functional derivation rules:

- Functional specs must describe observable behavior.
- Functional specs should not become module design documents.
- Technical implementation details belong in related technical docs unless users need the detail to understand behavior.
- Functional specs must distinguish implemented behavior, missing verification, and planned behavior.

Completion gate:

- Important user-visible workflows have functional specs or explicit deferrals.
- Every functional spec links related code and tests.
- Missing tests and behavior conflicts are recorded.

## Pass 5: Project README

Create or update the project README after the documentation set exists.

1. Point the README to the most important entry pages.
2. Include the project purpose, main runtime surfaces, setup and verification commands, documentation map, wiki entry point, and maintenance expectations.
3. Link the architecture document, functional index, high-level design index, module design index, project wiki, known defects, open decisions, and procedures.
4. Keep the README compact. It should route readers to deeper pages instead of duplicating them.
5. Do not write the README as a change log.

Completion gate:

- A new reader can find the project purpose, setup path, verification path, architecture entry point, functional entry point, technical entry point, and wiki entry point from the README.

## Project Wiki Integration

When the project uses docs/wiki, update the wiki after the documentation passes.

1. Treat generated functional and technical documents as wiki page subclasses when they live under docs/wiki.
2. Create or update hub and index pages that summarize the functional and technical documentation set.
3. Ensure durable wiki pages and subclasses include Related Code and Related Tests.
4. Link functional specs, architecture, high-level design documents, module design documents, source files, tests, procedures, known defects, and open decisions.
5. Record unresolved contradictions in wiki Open Questions.
6. Run wiki lint when wiki tooling is available.

The wiki summarizes and navigates. It does not outrank code, tests, functional specs, procedures, or technical design documents.

## Verification

After all passes:

1. Run the project's build command when it exists and the documentation work changed code, generated artifacts, or project metadata that can affect the build.
2. Run documentation or wiki lint when available.
3. Search the new documentation for unresolved TODO markers that are not intentional.
4. Search the new documentation for stale comparative language such as enhanced, revised, old, and new when the document is supposed to describe steady state.
5. Confirm every created document has related source, test, or source-not-yet-identified entries.
6. Confirm every Mermaid diagram represents a real sequence, association, aggregation, containment, ownership, dependency, lifecycle, coverage, or data movement relationship from the linked code, tests, and source documents.
7. Confirm every linked SVG diagram has an editable source and is marked as a rendered companion artifact.
8. Confirm every open question is specific enough for a human or future agent to answer.

## Stop Conditions

Stop and ask for human input when:

- The repository contains conflicting behavior that cannot be resolved from code, tests, and existing documents.
- A module boundary would require guessing business ownership.
- Functional behavior depends on unavailable private systems or data.
- Verification commands are required but cannot run for environmental reasons.
- The reverse engineering pass would need to send private or company information to an external service without explicit permission.
