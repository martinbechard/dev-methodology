# Development Documentation Methodology

## Purpose

This methodology defines one shared wiki-compatible page contract and five document specializations for software projects:

1. Project wiki page
2. Functional specification
3. Architecture
4. High-level design
5. Module design

The project wiki page contract is the base shape. Functional specifications, architecture documents, high-level designs, and module designs extend that base shape with sections needed for their domain. Functional specifications define user-visible behavior and acceptance expectations. Architecture defines the whole system and cross-cutting rules. High-level design decomposes a complete subsystem or feature area into cooperating pieces. Module design defines one implementation unit and its immediate context. The project wiki synthesizes the current understanding across functional documents, technical documents, code, tests, and project procedures.

## Wiki-Compatible Page Inheritance

Every durable project documentation page should start with the shared wiki-compatible sections:

- Current Understanding.
- Authoritative Sources.
- Related Code.
- Related Tests.
- Related Backlog Items.
- Related Wiki Pages.
- Open Questions.
- Maintenance Notes.

Specialized documents are page subclasses. A subclass keeps the shared sections first, then adds specialized sections after Maintenance Notes. The project wiki template defines the base contract and does not need to know every specialized document type. Each specialized template owns its own extension sections.

When a project keeps functional, architecture, high-level design, or module design documents under docs/wiki, the documents remain valid wiki pages because they inherit the shared page contract. When a project keeps the same documents outside docs/wiki, the inherited structure is still preferred because the documents can be moved into the wiki or linked from the wiki without changing their shape.

The shared sections keep navigation, authority, code references, test references, backlog references, cross-wiki links, unresolved questions, and maintenance history consistent across all documentation. The specialized extension sections keep domain detail out of the base wiki contract.

## Diagram Policy

Diagrams are optional documentation sections, not part of the mandatory base page contract. Add a diagram wherever a section defines multiple items whose meaning depends on sequence, association, aggregation, containment, ownership, dependency direction, lifecycle state, coverage, or data movement.

The diagram trigger is structural relationship, not prose weakness. If the section explains a list of related parts, a set of contained parts, a handoff sequence, a dependency stack, a lifecycle, a concern-to-owner map, or a verification coverage map, look for a compact diagram that shows the relationship directly.

Mermaid is the default editable source for diagrams because it lives with the Markdown page, can be reviewed in diffs, and can be regenerated by documentation tooling. Use SVG as an optional rendered artifact when the target wiki, review surface, exported deck, or static site cannot render Mermaid reliably, or when the visual needs layout fidelity beyond Mermaid. When both exist, the Mermaid source or another text diagram source is authoritative and the SVG is a generated companion. Link the SVG from the page and record its generation command or source relationship in Maintenance Notes.

Specialized templates own diagram placement:

- Functional specifications may include Workflow Diagram.
- Architecture documents may include Scope Boundary Diagram, System Context Diagram, Ownership Tree Diagram, Layered Dependency Diagram, Component Association Diagram, Data Flow Diagram, Lifecycle Diagram, Concern Ownership Map, and Verification Coverage Map.
- High-level designs may include Scope Boundary Diagram, Data Anchor Map, Constituent Component Association Diagram, Interaction Diagram, Lifecycle Diagram, Data Contract Map, Configuration Ownership Map, Implementation Sequence Diagram, and Verification Coverage Map.
- Module designs may include Processing Diagram.
- Project wiki topic pages may include a topic diagram only when the page owns a durable synthesized view.

Choose the diagram type from the relationship being documented:

- Sequence or handoff relationships use a sequence diagram by default. Use a flowchart only when branches, retries, or state decisions are the main relationship.
- Association relationships use a component association diagram.
- Aggregation or containment relationships use a tree or nested graph.
- Dependency relationships use a layered dependency diagram.
- Lifecycle relationships use a state diagram or ordered flowchart.
- Data movement relationships use a data flow diagram.
- Ownership or responsibility relationships use an ownership map.
- Verification relationships use a coverage map from behavior, layer, or component to tests.

Never replace source-linked prose with only a diagram. The page must still link code, tests, documents, and open questions. Completed documents should not include editorial notes that explain why a diagram was added or how the diagram was authored.

## Functional Specification

Functional specification documentation describes what users, administrators, operators, or external systems can do. It should answer who uses the workflow, where the workflow starts, what the user sees, which states and permissions matter, which edge cases must be handled, and how the behavior is verified.

Use functional specification documentation when a feature changes user-visible behavior, product workflow, admin workflow, route behavior, acceptance criteria, permission behavior, status display, operational affordance, or error state.

Functional specification documentation is a wiki page subclass. It starts with the shared wiki-compatible sections, then adds functional sections for:

- User or actor goal.
- Parent workflow and entry points.
- Route or surface list.
- Scope and non-goals.
- Concepts the user must understand.
- Workflow steps written from the user's point of view.
- States, rules, permissions, and edge cases.
- Verification blocks with test type, test files, status, scenario, steps, and assertions.
- Related architecture, high-level design, module design, tests, and wiki pages.

Generic examples:

- An admin approvals workflow that defines pending-user review, approve and deny actions, bulk actions, empty states, and related regression tests.
- A client onboarding workflow that defines sign-in, profile completion, plan selection, billing status, failure states, and support handoff.
- A report publication workflow that defines preview, validation, branch selection, publication snapshot creation, client visibility, and rollback limits.

Functional specs should live in the target project's functional documentation folder or under docs/wiki when the project uses the wiki as its documentation root. A common neutral pattern is [documentation-root]/functional. If a project already keeps functional specs under a product docs folder, keep that folder and document the convention.

Use templates/functional-spec-template.md for new functional specifications.

## Architecture

Architecture documentation describes the whole system or a cross-cutting system concern. It should answer what the system is, what technology it uses, how files are organized, how major layers communicate, and which design rules apply across many modules.

Use architecture documentation when a decision affects more than one subsystem, changes a shared project convention, defines a technical stack choice, or explains a cross-cutting concern such as state, errors, testing, configuration, object creation, persistence, security, privacy, observability, or UI composition.

Architecture documentation is a wiki page subclass. It starts with the shared wiki-compatible sections, then adds architecture sections for:

- System purpose and scope.
- Technology stack and runtime assumptions.
- File organization and key source paths.
- Major layers and dependency direction.
- Major components and their ownership.
- Data flow and lifecycle flow.
- Cross-cutting concerns and project-wide rules.
- Design principles and invariants.
- Risks, trade-offs, verification, and related documents.

Generic examples:

- A web application platform architecture that defines the frontend runtime, backend runtime, database choice, deployment model, folder structure, authentication approach, logging, testing, and dependency direction.
- A mobile synchronization architecture that defines offline storage, conflict resolution, sync lifecycle, retry behavior, privacy boundaries, and shared data ownership.
- An LLM workflow architecture that defines model providers, prompt storage, tool execution boundaries, persistence, evaluation, observability, and secret handling.

Architecture docs should live in the target project's architecture documentation folder or under docs/wiki when the project uses the wiki as its documentation root. A common neutral pattern is [documentation-root]/architecture, where [documentation-root] is whatever documentation root the target project already uses.

Use templates/architecture-template.md for new architecture documents.

## High-Level Design

High-level design documentation describes a complete subsystem, feature family, or system slice from the top down. It lists constituent modules, the order in which work should happen, the main interactions between pieces, and the boundaries between goals and non-goals.

Use high-level design documentation when a feature cannot be understood as one module, when implementation requires several cooperating modules, or when the main risk is coordination across services, UI, data, configuration, external integrations, or tests.

High-level design documentation is a wiki page subclass. It starts with the shared wiki-compatible sections, then adds high-level design sections for:

- Subsystem purpose and parent architecture.
- Scope and non-goals.
- Current data anchors and authoritative sources.
- Constituent components and their responsibilities.
- Interaction model, lifecycle, and data contracts.
- Configuration ownership.
- Implementation order.
- Cross-module invariants.
- Definition of good and verification.
- Related architecture, module, and test documents.

High-level design diagrams should live inside the section they clarify instead of becoming standalone sections. A constituent component diagram should show only components or explicitly named subsystem parts. Data artifacts, evidence records, configuration values, tests, and lifecycle states belong in their owning sections unless the diagram title makes the mixed relationship clear.

Generic examples:

- A checkout subsystem design that decomposes cart review, payment authorization, order persistence, receipt delivery, fraud checks, and fulfillment handoff.
- A file import subsystem design that decomposes upload intake, validation, parsing, transformation, duplicate detection, persistence, and user-visible import status.
- A notification subsystem design that decomposes event ingestion, preference filtering, template rendering, delivery adapters, retry policy, and audit logging.

High-level design docs should live in the target project's high-level design folder or under docs/wiki when the project uses the wiki as its documentation root. A common neutral pattern is [documentation-root]/high-level. If the design is a multi-document series, create a named folder with an index and ordered child documents.

Use templates/high-level-design-template.md for new high-level design documents.

## Module Design

Module design documentation describes one module, task, class, service, utility, UI component, or tightly scoped feature unit. It defines the module responsibility, runtime path, callers, dependencies, contracts, invariants, configuration, error handling, and verification.

Use module design documentation when creating or changing a particular implementation unit. A module design should be precise enough that tests and code can be written from it without guessing.

Module design documentation is a wiki page subclass. It starts with the shared wiki-compatible sections, then adds module design sections for:

- Purpose and runtime path.
- Parent context.
- Responsibilities.
- Callers and dependencies.
- Public contracts.
- Internal data and state.
- Processing rules.
- Invariants.
- Configuration, external interfaces, UI behavior, and notifications when applicable.
- Error handling.
- Verification.
- Related documents and source files.

Generic examples:

- A PaymentAuthorizationService module design that defines its responsibility, callers, gateway dependency, request and response contracts, retry rules, error handling, and verification.
- A CsvInvoiceParser module design that defines its accepted input shape, parsing rules, validation errors, output contract, and tests.
- A NotificationPreferencePanel module design that defines its UI responsibility, state inputs, save behavior, validation, accessibility expectations, and integration points.

Module design docs should live in the target project's module design folder or under docs/wiki when the project uses the wiki as its documentation root. A common neutral pattern is [documentation-root]/modules. Task designs may live under a module-design subfolder chosen by the target project.

Use templates/module-design-template.md for new module design documents.

## Project Wiki

Project wiki documentation is a synthesized knowledge layer. It helps future humans and agents find the right functional document, technical document, source file, test, procedure, backlog item, or open question without rereading the whole repository.

The wiki is not the highest source of truth. If the wiki conflicts with code, tests, functional specifications, procedures, architecture, high-level designs, module designs, or backlog status, update the wiki from those sources.

Use a project wiki when the project has more than a few functional or technical documents, when code behavior needs durable navigation, when recurring agents need current project context, or when daily commit activity should update project understanding.

Good project wiki documentation usually includes:

- README, schema, topic index, glossary, open decisions, known defects, and maintenance log.
- Functional topic pages for workflows, routes, roles, states, and acceptance behavior.
- Technical topic pages for architecture, subsystems, modules, data contracts, and operational rules.
- Code topic pages that map runtime surfaces to source files and tests.
- Related Code and Related Tests sections on durable topic pages.
- Source links to functional specs, technical docs, procedures, backlog items, code, and tests.
- Open Questions for unresolved contradictions or missing authority.

Project wikis should live in docs/wiki. Functional and technical documents may either stay in their established documentation folders and be linked from the wiki, or live in docs/wiki as wiki page subclasses. A wiki topic page should not silently replace a specialized document. When the specialized document is the maintained source document, it should use its subclass template and inherit the shared wiki-compatible sections.

Use templates/project-wiki-template.md when setting up a project wiki.

## Code Project Wiki Operating Model

A code-aware project wiki workflow should use the existing project wiki tooling as the wiki engine and add repository-specific code synchronization rules around it.

The recommended reusable workflow is a code-project-wiki skill. That skill should be a thin wrapper around the project-wiki skill. It should not duplicate project-wiki page quality rules, source priority rules, lint rules, or topic splitting rules.

The code-project-wiki workflow should:

- Initialize or normalize docs/wiki.
- Promote functional and technical documents into durable wiki page subclasses or durable wiki references.
- Ensure durable pages include Related Code and Related Tests.
- Link the source folder into the wiki root when local navigation benefits from it.
- Add the linked source path to .gitignore so the repository does not track the link.
- Treat daily commits as new source material to inspect.
- Update wiki pages only when commits change behavior, design intent, code ownership, verification, procedures, known defects, or open decisions.
- Record skipped commits with a reason when no wiki update is needed.
- Run wiki lint before completion.

The source link should be a local convenience, not a portable source citation. Wiki pages should link repository files using project-relative links where possible.

## Daily Commit Sync

Daily commit sync turns recent commits into wiki maintenance prompts. It should detect changes, read the authoritative files, and update the wiki only when project understanding changed.

The sync should:

1. Find commits after the last processed commit recorded in the wiki maintenance log or a small wiki state file.
2. Save a raw commit scan artifact with commit ids, messages, changed files, and initial classification.
3. Classify changed files as functional document, technical document, source code, test, configuration, migration, procedure, backlog, or generated artifact.
4. Read the changed functional documents, technical documents, source files, tests, and procedures before editing wiki pages.
5. Use project wiki suggestions to map changed files to likely wiki pages.
6. Update existing pages before creating new pages.
7. Create new durable pages only when the changed concept can recur or change independently.
8. Update Related Code and Related Tests when source ownership or verification changed.
9. Record open questions instead of guessing when code and documents disagree.
10. Run wiki lint and fix findings.
11. Record the processed commit range and changed wiki pages.

The daily sync must not treat commit messages as sufficient evidence. Commit messages are discovery hints. Code, tests, functional specs, procedures, and technical docs remain the authoritative material.

## Choosing The Right Document Type

Create a functional specification subclass when the question is what a user, admin, operator, or external system can do.

Create an architecture subclass when the decision shapes the whole project, a whole layer, or a cross-cutting concern.

Create a high-level design subclass when the work is one coherent subsystem but spans multiple modules or implementation phases.

Create a module design subclass when the work is one implementation unit whose behavior, dependencies, and tests can be specified directly.

Create or update project wiki pages when the project needs synthesized navigation, durable topic understanding, code references, test references, or commit-driven maintenance.

When a design spans document types, write the smallest complete set of documents:

- Functional specification defines the intended user-visible behavior.
- Architecture sets the global rule.
- High-level design applies the rule to a subsystem and decomposes the work.
- Module design gives each implementation unit its contract.
- Project wiki links, summarizes, and reconciles the current understanding.

Do not duplicate the same detail at every level. Link downward for detail and upward for context. The wiki may summarize, but it should point to the source documents and code that own the details.

## Applying The Methodology To A Project

For new projects:

1. Review the target project documentation.
2. Identify the strongest existing examples of functional specifications, architecture, high-level design, module design, and wiki pages.
3. Decide which project folder should be the documentation root.
4. Copy the templates into the project documentation area.
5. Copy this methodology into the project documentation area.
6. Replace [documentation-root] with the target project's chosen documentation root.
7. Replace every TODO in new documents with project-specific content.
8. Remove sections that are genuinely not applicable.
9. Add links to parent and child documents.
10. Verify the document reads as a steady-state design, not a comparison with an earlier version.
11. Use Mermaid as the editable diagram source when a section defines multiple items whose meaning depends on sequence, association, aggregation, containment, ownership, dependency direction, lifecycle state, coverage, or data flow.
12. Confirm that functional docs align with current user-visible behavior and regression tests.
13. Confirm that module docs align with the relevant source code, type registry, and file headers when the target project has those conventions.
14. Create or update docs/wiki so functional and technical knowledge is navigable from durable topic pages.
15. Configure code-aware wiki maintenance when daily commit activity should update project understanding.

For existing projects without a complete documentation set, use procedure-reverse-engineer-project-documentation.md. That procedure derives module design documents first, groups modules into high-level designs, derives architecture from the full technical set, derives functional specifications from the technical understanding, and then creates a README that points to the important entry pages.
