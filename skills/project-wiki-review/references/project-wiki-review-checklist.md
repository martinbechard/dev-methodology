# Project Wiki Review Checklist

## Purpose

Use this Review Checklist to verify a project wiki artifact created from the methodology templates.

## Shared Contract

- The artifact starts with Current Understanding, Authoritative Sources, Related Code, Related Tests, Related Backlog Items, Related Wiki Pages, Open Questions, and Maintenance Notes.
- Current Understanding summarizes durable project knowledge instead of narrating edit history.
- Authoritative Sources identifies which source wins when sources disagree.
- Related Code and Related Tests link project-relative evidence or say Not yet identified after a real search.
- Open Questions record unresolved authority, ownership, sync cadence, automation, source-link, or page granularity issues.
- Maintenance Notes explain what future maintainers must recheck.

## Artifact-Specific Checks

- Wiki Root confirms docs/wiki or explains the project-specific exception.
- Authority Order includes code, tests, functional specs, procedures, architecture, high-level designs, module designs, backlog, and generated documentation as applicable.
- Base Page Contract names the shared sections and applies them to durable wiki pages and specialized documentation pages.
- Page Subclassing explains how specialized documents extend the base contract without changing it.
- Diagram Policy allows diagrams only for real sequence, ownership, dependency, lifecycle, state, data flow, or coverage relationships.
- Topic Pages define broad topic pages, folder indexes, leaf pages, and links to specialized documents.
- Code Pages map source folders, test folders, scripts, configuration, migrations, and generated artifact sources when those are meaningful.
- Local Source Link describes whether a local source link exists, how it is ignored by git, and why durable links remain project-relative.
- Daily Commit Sync defines range source, raw scan location, file classification, skip recording, and the rule that commit messages are discovery hints.
- Update Workflow includes status, changed-file suggestion, source reading, page update, related code update, related test update, open question handling, lint, and maintenance log update.
- Automation names scheduled or manual maintenance flows and what each may write.
- Verification lists wiki status, lint, page review, link checks, and sample sync checks.

## Findings

Report findings first. Treat missing source authority, missing Related Code, missing Related Tests, unresolved TODO markers, broken links, and unsourced automation claims as review findings.
