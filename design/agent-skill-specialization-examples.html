Agent Skill Specialization Examples

These customer-safe examples use fictitious projects to show how AGENTS-PLAN.md organizes role agents, skill loadouts, nested routing, and validation evidence. The generic strategy lives in Agent And Skill Specialization Strategy.

Standalone Planning Artifact

AGENTS-PLAN.md

AGENTS-PLAN.md is the project-owned setup document for validating agent roles and skill routing before durable guidance is copied into AGENTS.md. It can live at the repository root and, when needed, inside technology-specific subfolders.

Project Agent Setup Agent

This setup role uses development-methodology and create-agents-plan to inspect the project, classify tiers, choose skill loadouts, and create the root and nested AGENTS-PLAN.md files.

Private Validation

For proprietary projects, create AGENTS-PLAN.md inside the private project folder. Keep project names, internal workflows, private architecture, customer data, and validation notes inside that repository.

Customer Examples

Customer-facing examples use fictitious product names, synthetic paths, and generic behavior. They show the routing pattern without leaking private project details.

AGENTS-PLAN.md Template Sections

Section Purpose How it validates routing
Current Understanding Summarizes the project purpose, stage, and reason agent routing is needed. Prevents a role map from being created without a source-backed project boundary.
Authoritative Sources Lists README files, AGENTS.md files, procedures, packages, tests, docs, and source roots inspected. Makes the setup reviewable because every role and skill decision can be traced to evidence.
Project Taxonomy Classifies project family, application tiers, technology stacks, runtime boundaries, and documentation surfaces. Gives the skills router stable terms such as web application, front-end tier, API tier, static pages tier, DB tier, documentation tier, or CLI tier.
Root AGENTS.md Routing Reference Explains what the operational AGENTS.md file should tell agents when they enter the repository. Separates planning evidence from the concise routing instructions agents load during normal work.
Role Agent Set Defines each project role, the work it owns, handoff boundaries, loaded skills, and completion checks. Shows when a reusable role is enough and when a specialist reviewer or verification agent is justified.
Skill Loadouts Maps each tier, technology, workflow, or path family to reusable skills. Keeps skill specialization factual and avoids turning every project behavior into a one-off agent.
Folder Routing Maps folders and path patterns to roles, tiers, skills, source evidence, and verification commands. Lets the router decide quickly whether a task is front-end, API, DB, docs, testing, or tool work.
Nested AGENTS-PLAN Files Lists subfolders that need their own local plan because they have distinct technology or verification rules. Keeps root guidance compact while allowing deeper project areas to document local loadouts.
Validation Plan Defines commands and checks used to prove the role and skill setup is usable. Turns the plan into a standalone validation artifact for private projects and customer deployments.

Private Project Validation Flow

Run this flow separately inside each private project. The distributable bundle keeps only the template, setup skill, and fictitious examples; the project-specific AGENTS-PLAN.md files stay with the private repositories they describe.

Step What the setup agent inspects Output Validation question
Inventory README files, AGENTS.md files, package metadata, source roots, tests, docs, wiki pages, and procedures. Current Understanding and Authoritative Sources sections. Does the plan describe the actual project rather than a remembered project shape?
Classify Runtime folders, application tiers, persistent data, generated artifacts, and documentation surfaces. Project Taxonomy and Folder Routing sections. Can the skills router identify front-end, API, static pages, DB, docs, test, or CLI work from paths?
Assign Reusable roles, project-specific verification needs, privacy constraints, and handoff points. Role Agent Set and Skill Loadouts sections. Are specialist roles justified by distinct output contracts rather than by folder names alone?
Nest Subfolders with different technology, runtime setup, data handling, or verification commands. Nested AGENTS-PLAN.md files where local detail is valuable. Does nesting reduce cognitive load without hiding repository-wide rules?
Verify Build commands, test commands, docs checks, wiki checks, and page verifier results. Validation Plan, Open Questions, and Maintenance Notes sections. Can another agent use the plan to update AGENTS.md and load the right skills?
