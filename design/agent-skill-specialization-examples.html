Agent Skill Specialization Examples

This page owns three customer-safe configuration examples: a root-only tool, a multi-tier application with nested guidance, and a wiki repository with separate source and synthesis workflows. The strategy owns selection policy, while the operating model owns setup sequence and file responsibilities.

Configuration Examples

Each example contains one focused PROJECT.yaml excerpt and only the decision that distinguishes it. The canonical template owns the complete schema.

Northwind Tools: Root-Only Guidance

A single TypeScript runtime and common test workflow do not justify nested AGENTS.md files.

schema: project
schema_version: 1
project_taxonomy:
  family: local-developer-tooling
  primary_runtime: Node.js
  application_tiers:
    - name: CLI and file processing
      paths: [src/**]
      stacks: [TypeScript]
root_agents_reference:
  file: AGENTS.md
nested_agents_files: []
technology_skill_loadouts:
  - scope: source and tests
    pathPattern: "{src,test,tests}/**"
    skills: [node-cli, typescript-esm, typescript-strict, vitest]
validation_plan:
  commands:
    - command: pnpm test
      proves: command and fixture behavior
    - command: pnpm run build
      proves: package compilation

Acme Ledger: Nested Tier Guidance

Repository-wide rules stay at the root; only API contracts and database migration rules require nested guidance.

schema: project
schema_version: 1
project_taxonomy:
  family: web-application
  application_tiers:
    - name: Front-end
      paths: [app/(authenticated)/**, components/**]
      stacks: [Next.js, React]
    - name: API
      paths: [app/api/**]
      stacks: [Next.js routes]
    - name: DB
      paths: [db/**, drizzle/**]
      stacks: [Postgres, Drizzle]
root_agents_reference:
  file: AGENTS.md
nested_agents_files:
  - path: app/api/AGENTS.md
    needed: true
    reason: shared authentication and request-contract checks
  - path: db/AGENTS.md
    needed: true
    reason: migration, rollback, and sensitive-data rules
technology_skill_loadouts:
  - scope: API
    pathPattern: app/api/**
    skills: [api-routes, clerk-auth, postgres-drizzle]
  - scope: DB
    pathPattern: db/**
    skills: [postgres-drizzle, typescript-strict, jest]

Beacon Knowledge Base: Workflow Separation

One root guidance file is sufficient, but its routing keeps raw collection separate from durable wiki maintenance and ingest.

schema: project
schema_version: 1
project_taxonomy:
  family: documentation-and-project-wiki
  application_tiers:
    - name: Source collection
      paths: [raw/inbox/**]
      stacks: [Public web sources]
    - name: Ingest queue
      paths: [raw/Clippings/**]
      stacks: [Queued Markdown sources]
    - name: Wiki synthesis
      paths: [docs/wiki/**]
      stacks: [Markdown project wiki]
root_agents_reference:
  file: AGENTS.md
nested_agents_files: []
folder_routing:
  - pattern: raw/inbox/**
    tier_or_workflow: raw-only source collection
    primary_role: Wiki Source Collector
    required_skills: [project-wiki-research]
    verification_commands: [wiki_ops.py status]
  - pattern: raw/Clippings/**
    tier_or_workflow: raw-to-durable ingest
    primary_role: Wiki Ingester
    required_skills: [project-wiki]
    verification_commands: [wiki_ops.py status]
  - pattern: docs/wiki/**
    tier_or_workflow: ordinary durable wiki maintenance
    primary_role: Wiki Writer
    required_skills: [project-wiki, project-wiki-topic-write]
    verification_commands:
      - wiki_ops.py lint
      - wiki_ops.py okf-validate
