Agent Skill Specialization Examples

These customer-safe examples use fictitious projects to show how AGENTS-PLAN.yaml organizes role agents, skill loadouts, nested routing, setup evidence, and verification checks. AGENTS-PLAN.yaml is the checked-in YAML planning artifact; the samples below show the file content directly. The generic strategy lives in Agent And Skill Specialization Strategy.

Standalone Planning Artifact

AGENTS-PLAN.yaml

AGENTS-PLAN.yaml is the project-owned setup document for planning agent roles and source-backed folder technology loadouts before durable guidance is copied into AGENTS.md. It can live at the repository root and, when needed, inside subfolders with distinct runtime or verification boundaries.

Project Agent Setup Agent

This setup role uses development-methodology, create-agents-plan, and detect-technology-skills to inspect the project, classify tiers, detect variants for representative folder scopes, record source evidence, and create the root and nested AGENTS-PLAN.yaml files.

Project-Owned Setup

Create AGENTS-PLAN.yaml inside the project folder it describes. Public samples, internal projects, and customer deployments follow the same setup flow; only the source evidence and local rules change.

Customer Examples

Customer-facing examples use fictitious product names, synthetic paths, and generic behavior. They show the routing pattern without depending on real project details.

AGENTS-PLAN.yaml Template Sections

Section Purpose What it establishes
Current Understanding Summarizes the project purpose, stage, and reason agent routing is needed. Prevents a role map from being created without a source-backed project boundary.
Authoritative Sources Lists README files, AGENTS.md files, procedures, packages, tests, docs, and source roots inspected. Makes the setup reviewable because every role and skill decision can be traced to evidence.
Project Taxonomy Classifies project family, application tiers, technology stacks, runtime boundaries, and documentation surfaces. Gives Project Agent Setup stable terms such as web application, front-end tier, API tier, static pages tier, DB tier, documentation tier, or CLI tier.
Root AGENTS.md Routing Reference Explains what the operational AGENTS.md file should tell agents when they enter the repository. Separates planning evidence from the concise routing instructions agents load during normal work.
Role Agent Set Defines each project role, the work it owns, handoff boundaries, loaded skills, and completion checks. Shows when a reusable role is enough and when a specialist reviewer or verification agent is justified.
Technology Skill Loadouts Maps each folder scope to reusable technology skills with source evidence, required-load policy, and missing or conflicting results. Keeps skill specialization factual and avoids turning every project behavior into a one-off agent.
Folder Routing Maps folders and path patterns to roles, tiers, skills, source evidence, and verification commands. Lets every agent load the established folder skills without rerunning setup-time detection.
Nested AGENTS.md And AGENTS-PLAN.yaml Files Lists subfolders that need nested operational AGENTS.md references, and notes when a subtree also needs its own local AGENTS-PLAN.yaml because it has distinct evidence. Keeps root guidance compact while making local loadouts and setup evidence explicit.
Validation Plan Defines commands and checks used to confirm the role and skill setup is usable. Makes the setup plan verifiable without making validation the reason the plan exists.

Northwind AGENTS-PLAN.yaml

This is the root YAML plan for a simple tooling repository.

schema: agents-plan
schema_version: 1
project:
  name: Northwind Tools
  family: local-developer-tooling
  application_tiers: [cli, file-processing, generated-artifacts, tests, documentation]
root_agents_reference:
  file: AGENTS.md
  purpose: short router contract for CLI, generated artifacts, and tests
nested_agents_reference:
  required: false
  reason: one TypeScript runtime and one verification workflow
technology_skill_loadouts:
  - scope: CLI source
    pathPattern: src/**
    skills: [node-cli, typescript-esm, typescript-strict, vitest]
    sourceEvidence:
      - skill: node-cli
        evidence: [scope path **/commands/**: src/commands/build.ts]
validation:
  commands: [pnpm test, pnpm run build]

Acme AGENTS-PLAN.yaml

This root plan points to nested operational AGENTS.md references for API and database local rules.

schema: agents-plan
schema_version: 1
project:
  name: Acme Ledger
  family: web-application
operational_references:
  root: AGENTS.md
  nested: [app/api/AGENTS.md, db/AGENTS.md]
application_tiers:
  front_end:
    paths: [app/(authenticated)/**, components/**]
    skills: [nextjs-app-router, react-server-components, tailwind-design-system]
  api:
    paths: [app/api/**/route.ts]
    skills: [api-routes, clerk-auth, postgres-drizzle]
    nested_agents_reference: app/api/AGENTS.md
  static_pages:
    paths: [app/(marketing)/**]
    skills: [nextjs-app-router, react-server-components, playwright]
  db:
    paths: [db/**, drizzle/**]
    skills: [postgres-drizzle, typescript-strict, jest]
    nested_agents_reference: db/AGENTS.md

dispatch:
  project-agent-setup-agent:
    skills: [development-methodology, create-agents-plan, documentation-page-verifier]
  coding-agent:
    rules:
      - match: app/api/**/route.ts
        add_skills: [api-routes, clerk-auth, postgres-drizzle, jest]
      - match: db/**
        add_skills: [postgres-drizzle, typescript-strict, jest]

Project Setup Flow

Run this flow on the customer machine before writing or refreshing project guidance. The setup agent installs the selected runtime adapter, creates AGENTS-PLAN.yaml, chooses where nested guidance belongs, emits AGENTS.md and thin CLAUDE.md bridges where needed, and records evidence.

Step What the setup agent inspects Setup output Completion check
Install Detected runtime, generated native agent package, generic skills, support scripts, and destination scope. Owned user-scope skills and native agent definitions. The runtime discovers the installed skills and agents, and generic assets contain no customer material.
Inventory README files, AGENTS.md files, package metadata, source roots, tests, docs, wiki pages, and procedures. Current Understanding and Authoritative Sources sections. The plan describes the actual project rather than a remembered project shape.
Classify Runtime folders, application tiers, persistent data, generated artifacts, and documentation surfaces. Project Taxonomy and Folder Routing sections. Project Agent Setup can identify separate front-end, API, static pages, DB, docs, test, or CLI folder scopes.
Assign Reusable roles, project-specific verification needs, privacy constraints, and handoff points. Role Agent Set and Skill Loadouts sections. Specialist roles are justified by distinct output contracts rather than by folder names alone.
Place Subfolders with different technology, runtime setup, data handling, or verification commands. Nested AGENTS.md references, thin colocated CLAUDE.md imports, and local AGENTS-PLAN.yaml files where local detail is valuable. Nested guidance reduces cognitive load without hiding repository-wide rules or duplicating Claude guidance.
Verify Build commands, test commands, docs checks, wiki checks, and page verifier results. Validation Plan, Open Questions, and Maintenance Notes sections. Another agent can use the plan to update AGENTS.md and load the right skills.

Example: Direct Customer Customization

A customer edits an installed code-review skill to add audit evidence rules while keeping the skill name stable. Existing agents and AGENTS.md references continue to resolve it.

During a later update, an agent compares the old generic skill, the installed customer skill, and the new generic skill. The customer chooses which differences to retain, merge, replace, or remove before any file is updated.
