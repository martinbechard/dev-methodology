Agent Skill Specialization Examples

These examples apply the shared agent and skill specialization strategy to two concrete repositories. The generic strategy lives in Agent And Skill Specialization Strategy.

gemma-chat-public

Project classification

  • Family: local-model experimentation app, Electron desktop app, custom harness.
  • Tier: greenfield experimental product with local-first verification.
  • Primary stacks: TypeScript, Electron, Vite, React, Node, Vitest.
  • Application domains: custom harness, product tool runtime, plan engine, local MLX serving, configured remote model endpoints.

Boundary rule

Gemma's own harness, tools, planning loop, prompts, and local model runtime are product internals. Development agents load skills for those domains when they code, review, test, or document the project; they are not counted as methodology agents that a human invokes unless a separate development context or output contract is needed.

Work target Coding Agent dispatch Verification dispatch Specialized agent?
src/renderer/** Load react-vite-renderer, electron-preload, typescript-esm, and vitest. Apply project renderer-state and accessibility rules from the profile. Run focused renderer Vitest tests, then full tests and build for completed code work. No. This is a frontend skill route inside Coding Agent.
src/main/** Load electron-main, electron-preload, node-cli, typescript-esm, and local-model-integration when model routing is touched. Run focused main tests, typecheck or build, and full test suite when behavior changes. No. Main-process work is a stack and runtime route.
src/main/tools/** Load tool-runtime-implementation, tool-contract, prompt-surface, action-parser, protected-file, and tool registry skills. Run tool tests and prompt tests such as action parser, tool layout, and code system prompt coverage. No for normal tools. The tool runtime is product code. A dedicated Tool Contract Reviewer is useful for independent review only.
src/main/plan/** Load plan-engine-implementation, YAML parsing, semantic review, evidence, retry policy, and verification prompt skills. Run focused plan tests and CLI plan prompt tests. No for implementation. The plan engine is product code. Code Review Agent stays separate for fresh-context review.
Gemma*.md Load harness-implementation, tool-runtime-implementation, and plan-engine-implementation. Apply prompt contract rules from the project profile. Run prompt-related tests and inspect affected mode behavior. Sometimes. A Prompt Contract Reviewer is useful when prompt safety and tool protocol need independent review.
MLX runtime Load local-model-integration, Python environment, cache, server lifecycle, and runtime diagnostics skills. Run MLX-focused tests or smoke checks where local resources allow. Sometimes. Runtime Diagnostician is a development support agent for setup failures and logs, not a Gemma runtime agent.
docs/wiki/** Route to Documentation Architect or Wiki Ingest Agent instead of Coding Agent. Run wiki status, lint, and relevant link checks. Yes. Documentation and wiki synthesis have different output contracts.

mpact-insight-questionnaire

Project classification

  • Family: full-stack web product, API service, analysis workflow, documentation-rich application.
  • Tier: greenfield product with production-style verification and no legacy compatibility layer.
  • Primary stacks: Next.js 15, React 19, TypeScript, PostgreSQL, Drizzle, Clerk, Tailwind, Jest, Playwright.
  • Special runtime: LangGraph analysis flows, Vercel AI Gateway, QStash, chatbot RAG, Dev versus QA runtime boundary.

Work target Coding Agent dispatch Verification dispatch Specialized agent?
src/app/** page or layout Load Next.js App Router, React Server Components, TypeScript, auth boundary, and UI composition skills. Run relevant Jest tests, build, and smoke checks for routing, layout, middleware, or auth changes. No. This is a Coding Agent route.
src/app/api/**/route.ts Load api-routes, clerk-auth, nextjs-app-router, typescript-strict, postgres-drizzle, and jest. Run API unit or integration tests, build, and affected E2E tests when user-visible. No. API language and framework knowledge is a skill route.
src/components/** Load react-server-components, tailwind-design-system, typescript-strict, and playwright when workflow coverage is affected. Run component tests and E2E tests for changed user workflows. No for coding. UX Reviewer remains separate for independent design review.
src/lib/server/** Load typescript-strict, postgres-drizzle, langgraph, clerk-auth, and jest based on the touched server subfolder. Run focused Jest tests, build, and integration tests for persistence or workflow changes. No. Server area changes are routed skills unless they need a separate verifier.
src/prompts/** Load langgraph, typescript-strict, and jest. Apply project prompt, fixture, RAG, and data-safety rules from the profile. Run prompt fixture tests, index generation checks, and affected analysis tests. Sometimes. Prompt Review Agent is useful for independent prompt safety or evaluation review.
tests/*.spec.ts Route to E2E Browser Agent when creating or fixing Playwright workflows, because browser state and runtime resources are exclusive. Run targeted Playwright command against Dev, not QA, with console error checks. Yes. E2E work owns a distinct runtime and output contract.
docs or docs/wiki Route to Documentation Architect or Wiki Ingest Agent for specs, project wiki, chatbot RAG docs, and user-facing docs. Run wiki lint and docs-specific checks, plus build when docs feed generated app artifacts. Yes. Documentation has a distinct artifact contract.
