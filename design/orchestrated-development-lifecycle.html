Path decision Dev Orchestrator One accepted contribution lane keeps its independently reviewed and verified commit as the final direct commit, without Dev Merge Coordinator. Multiple accepted committed contributions enter the multi-lane integration path.
Multi-lane integration, when required Dev Merge Coordinator Combine multiple accepted commits in dependency order, with their claim, review, and verification evidence. Do not create a merge lane for one accepted contribution.
Integrated-artifact review and verification, multiple lanes only Appropriate task-selected independent reviewers, then Dev Verifier, each in fresh contexts After combination, every changed surface is reviewed in a fresh read-only context by its appropriate reviewer or reviewers: source changes by Dev Code Reviewer, and non-source setup, documentation, and wiki artifacts by their task-selected independent artifact or domain reviewers. All applicable post-integration review gates must pass before Dev Verifier checks the complete integrated outcome in another fresh context. Both stages must pass before the result becomes the final integration commit.
Correction or blocker Original executor, Dev Merge Coordinator, and Dev Orchestrator Implementation findings and failed behavior checks return to the original executor; integration-only findings and conflict defects return to Dev Merge Coordinator. Reconstruction Evaluation Boundary

Pass 6 proves that the documentation package is an executable construction contract, not only a navigable explanation of the source.

Original baseline and seed

The reset source inventory is reconciled path by path to MUST_DOCUMENT, PUBLIC_GENERATOR, or PARITY_TEST_ONLY. The original oracle is captured before builder isolation. A brand-new destination receives docs/wiki first, then the complete documentation, routing, public-generator, and parity corpus. Relative source links remain recorded evidence references and never copy application source. Workstation user-home and checkout paths are rejected while declared container roots remain portable runtime inputs.

Enforced builder

Dev Coder receives only the validated seed and declared external toolchain through an operating-system sandbox or container. The exact production profile first proves actual byte-read denial through independent readers and agent-claim acquire/release in a disposable repository. Mount, environment, cache, temporary-path, network, credential, dependency-provenance, and denied-access evidence is recorded in the contamination ledger; permission predicates alone are not evidence.

Independent verifier

A distinct fresh Dev Verifier receives the reconstruction, immutable oracle, parity cases, and generator contracts without original source or builder-private context. For this route it loads the reconstruction and prompt-contract skills in addition to its normal verification skills. One digest seals the evaluator, contract, inventories, and oracle. Verifier-derived artifact, process, command, root, and run provenance binds every result. The evaluator rejects unknown, unattempted, weak, partial, synthetic, skipped, or unmatched-failure evidence and recomputes exact case-ID and generated-file reconciliation across build, tests, routes, workflows, data, security, integrations, configuration, operations, and deployment.

Review and archive

Dev Artifact Reviewer re-runs deterministic checks with the reconstruction-readiness checklist. New archives use schema version 2 and fail on hollow machine evidence, duplicate manifest paths, incomplete documentation hierarchy, seed/configuration drift, stale native proof, missing raw browser-report provenance, missing full pre/post Git metadata reconciliation, or insufficient builder and verifier isolation. A detached sibling attestation binds the frozen manifest and content aggregate. Existing schema-version-1 archives remain readable but cannot be newly sealed. The new archive and sidecar validate before retention pruning, after which the newest three runs remain unless the user selected another count.

Checklist model evaluation

The reconstruction-review corpus seals one synthetic artifact, its source contract, checklist version, and adjudicated defect set under one corpus ID. Candidate and reference reviewers run separately at least three times. The offline runner derives recall, false positives, checklist and citation completeness, elapsed time, tokens, and available cost without invoking a model or changing routing. Promotion requires no quality regression and at least twenty percent lower median time or tokens without increasing the other measure.

Execution Evidence

An orchestrated run leaves four complementary records. Each fact has one record owner so later readers do not have to reconcile competing summaries.

Record Owns Does not own
.agents/runs/<run-id>/context-pack.json Inputs supplied to the assigned agent: files, applicable guidance, constraints, and selected skills. Results or verification claims.
.agents/runs/<run-id>/agent-results.jsonl Ordered agent outputs and status events for replay, audit, and merge coordination. A synthesized verification verdict.
.agents/runs/<run-id>/verification.md Checks run, outcomes, unresolved risks, and justified omissions. Task history or next-owner context.
.agents/runs/<run-id>/handoff.md Completed work, remaining work, follow-up ownership, and the context needed to continue. Raw event history or duplicated command transcripts.
