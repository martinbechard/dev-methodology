Orchestrated Development Lifecycle

This page describes how the toolkit's agents and skills can be combined into longer workflows to achieve more autonomous agency across bootstrap, planned design, complete source-backed documentation, execution, review, verification, integration, completion, and evidence capture. Use it after the individual agents have been calibrated through manual invocation and their results provide confidence that combining them into a longer workflow is likely to succeed.

Orchestrated Development Loop

Step Owner Transition or gate
Bootstrap, when requested Project Bootstrapper Reuse a valid project-root PROJECT.yaml. When it is absent, stale, or explicitly selected for reconfiguration, Project Configurator must create and validate it before documentation work begins. Whole-project reverse engineering reconciles the exact path inventory, reviews every meaningful module, groups the complete set into reviewed high-level designs, derives reviewed architecture and functional specifications, and verifies README/wiki navigation. Each level is a stop gate for the next; the accepted navigation gate completes reverse engineering.
Intake Dev Orchestrator Establish the task boundary, assignments, and validation route before handing work to the selected agents.
Claim gate Every modifying claim owner Mutation begins only after agent-claim establishes ownership of the affected files and exclusive resources.
Placement, when needed Project Organiser A new artifact proceeds only after its repository path is approved.
Execution Selected implementation or writing agent Each assigned lane stays within its narrow claim and returns a committed, clean handoff before independent review; the generated catalog owns each conceptual agent definition's detailed inputs, skills, and output contract.
Independent contribution review Dev Code Reviewer for source changes; task-selected independent artifact or domain reviewers for setup, documentation, and wiki changes Every changed surface goes to its appropriate reviewer in a fresh read-only context. Source changes go to Dev Code Reviewer; non-source setup, documentation, and wiki artifacts go to their task-selected independent artifact or domain reviewers. A reviewer accepts the contribution or returns correctable findings to the original executor. A correction re-enters fresh review within the bounded correction loop; after all applicable contribution-review gates pass, the accepted lane advances to Dev Verifier.
Contribution verification Dev Verifier The acceptance checks required by the behavior and risk must pass without weakened gates. A failed behavior check returns to the original executor; an accepted lane advances to the path decision.
Path decision Dev Orchestrator One accepted contribution lane keeps its independently reviewed and verified commit as the final direct commit, without Dev Merge Coordinator. Multiple accepted committed contributions enter the multi-lane integration path.
Multi-lane integration, when required Dev Merge Coordinator Combine multiple accepted commits in dependency order, with their claim, review, and verification evidence. Do not create a merge lane for one accepted contribution.
Integrated-artifact review and verification, multiple lanes only Appropriate task-selected independent reviewers, then Dev Verifier, each in fresh contexts After combination, every changed surface is reviewed in a fresh read-only context by its appropriate reviewer or reviewers: source changes by Dev Code Reviewer, and non-source setup, documentation, and wiki artifacts by their task-selected independent artifact or domain reviewers. All applicable post-integration review gates must pass before Dev Verifier checks the complete integrated outcome in another fresh context. Both stages must pass before the result becomes the final integration commit.
Correction or blocker Original executor, Dev Merge Coordinator, and Dev Orchestrator Implementation findings and failed behavior checks return to the original executor; integration-only findings and conflict defects return to Dev Merge Coordinator. A corrected contribution or combined artifact re-enters the applicable fresh review and verification gates. After two failed correction attempts for the same finding or acceptance criterion, or when a required fixed dependency is unavailable, Dev Orchestrator reports BLOCKED with preserved commits and evidence rather than substituting self-review.
Completion and claim release Dev Orchestrator and every claim owner Record the final direct commit for the single-lane path, or the final integration commit after integrated review and verification for the multi-lane path. Release ownership only after clean status and resource cleanup or handoff.

Planned Design Progression

Normal development uses the shared production skills from accepted functional intent down to verified code. Project-specific evaluation harnesses may compare frozen outputs afterward, but hidden reference artifacts, evaluator rubrics, and an original implementation do not enter production generation.

Stage Production input and owner Acceptance gate
Functional intent Accepted functional specifications and architecture define required outcomes, actors, permissions, workflows, constraints, and system boundaries. Every applicable requirement has stable authority or is explicitly open before subsystem design begins. A primary and supporting operation inventory preserves every directly invoked route, API, command, event, job, notification, and reference-data lookup with its observable access, selector, paging, projection, response, error, state, and verification contract.
High-level design The standard create-high-level-design skill works in PLANNED_DEVELOPMENT mode; review-high-level-design independently checks the result. Requirements coverage, critical trust and identity boundaries, and cross-module contracts pass review. A source-category inventory, operation-and-obligation inventory, and boundary-edge inventory close before prose so scheduled work, source open questions, caches, privileged tasks, and every producer-consumer edge remain visible. Each exact method, route variant, supporting UI action, command, event, or job identity remains one-to-one operation-level coverage; an abstract group cannot replace it. Each applicable anonymous, authenticated, administrator, service, or background actor receives an explicit boundary row, including denial or equivalent-role behavior. Local links resolve from the installed documentation path and never cite transient assembly or control files. An unresolved high-impact actor, authority, selector, disclosure, validation, state, transaction, asynchronous, or error-timing contract blocks downstream module design.
Module design The standard create-module-design skill consumes the accepted functional and HLD inputs; review-module-design independently checks the result. The candidate first passes an exact ordered template-heading and readiness-marker gate; missing, renamed, duplicated, merged, or reordered mandatory sections, or an Implementation Readiness section that does not begin with READY or BLOCKED, stop the candidate before semantic review. Local links resolve from the installed documentation path and never cite transient assembly or control files. A source-category inventory and owning-HLD open-question handoff close before prose. Every applicable requirement then maps to an implementable contract, rule, state, error path, and verification obligation. Scope-bearing qualifiers from the target assignment and owning-HLD component description remain explicit operation facets rather than being shortened to a generic module label. Compatible facts from accepted inputs are reconciled for the same exact operation without borrowing sibling contracts. Identity selectors, validation ownership, response disclosure, returned or emitted values, persistent state and cache mutations, side-effect phase ownership, failure timing, and sensitive logging are explicit and internally consistent. Executor acceptance precedes executor-owned work, while separately supported initiator precomputation and later sender or provider rejection remain distinct. Labels such as public do not substitute for operation-specific authentication evidence.
Implementation The ordinary coding agent implements the accepted design with the technology skills already routed for the affected project folders. Implementation does not receive project-specific reconstruction instructions or evaluator guidance. It preserves accepted contracts and supplies tests for the named verification obligations.
Verification and evaluation Normal independent review and verification prove the implementation. A separate project-owned evaluator may then compare frozen artifacts or observable behavior with hidden references. Production acceptance remains distinct from comparative scoring. The evaluator records fidelity or quality findings without changing the generation instructions after seeing the answer key.

Execution Evidence

An orchestrated run leaves four complementary records. Each fact has one record owner so later readers do not have to reconcile competing summaries.

Record Owns Does not own
.agents/runs/<run-id>/context-pack.json Inputs supplied to the assigned agent: files, applicable guidance, constraints, and selected skills. Results or verification claims.
.agents/runs/<run-id>/agent-results.jsonl Ordered agent outputs and status events for replay, audit, and merge coordination. A synthesized verification verdict.
.agents/runs/<run-id>/verification.md Checks run, outcomes, unresolved risks, and justified omissions. Task history or next-owner context.
.agents/runs/<run-id>/handoff.md Completed work, remaining work, follow-up ownership, and the context needed to continue. Raw event history or duplicated command transcripts.
