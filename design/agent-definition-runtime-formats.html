Agent Definition Runtime Formats

This page defines the common information needed to describe reusable methodology roles once, validate those roles against the skill catalog, render them into native runtime files, and expose them to the HTML role map without a second hand-written loadout source.

Role source Runtime adapters Skill validation Drift control

Current Understanding

Single Source Of Truth

Bundled skills remain sourced from skills/*/SKILL.md. Role definitions should live in real role files and list only skill IDs that exist in the generated skill catalog.

Generated Role Data

agent-role-skill-map.html should not hardcode role loadouts. It should read generated role data that was produced from role source files and validated against generated skill definitions.

Runtime Differences

Codex uses one standalone TOML configuration file per custom agent type. Claude Code, Gemini CLI, Junie CLI, and GitHub Copilot use Markdown files with YAML metadata or frontmatter. Field names and skill-loading semantics differ enough that one native file cannot serve every runtime.

Properties We Use

The canonical role source uses portable role properties. Semantic model profiles remain provider-neutral; runtime adapters resolve them to native model and reasoning fields.

Element Canonical Property Role In A Reusable Agent Definition
Name name Names the agent type used by routing, references, generated files, and UI display. Use a stable, short name that can survive adapter generation.
Filename filename Controls discovery and precedence in runtimes that load agent files from scoped folders. Keep it aligned with the name unless the runtime explicitly allows a different display name.
Description description Explains the agent's purpose and when a parent agent or user should delegate to it. This is a routing field, so it should be concise and specific.
Instructions instructions Defines behavior, boundaries, workflow, deliverables, and handoff expectations. Use this for portable role behavior when a runtime has no native field for a concern.
Skills skills Names the role's generic skill set. An entry without condition is fixed and always loads. An entry with condition is request-specific and is loaded only when the agent judges that condition applicable. Optional skillAvailability entries remain name-based or path-based Codex availability overrides, not routing conditions.
Skill condition skills[].condition Provides a human-readable when ... condition beneath one skill entry. The generator writes Use the skill-name skill when condition into runtime instructions and tells the agent to inspect the requested outcome and evidence when request wording is ambiguous.
Tools tools Lists tools or tool groups the agent may use. Some runtimes treat this as an allowlist; some default to all tools when omitted.
Disallowed tools disallowedTools Lists tools or tool groups the agent must not use. Keep this separate from tools because allowlists and denylists resolve differently across runtimes.
MCP servers mcpServers Names or configures MCP servers available to the agent. Keep server access separate from tool names because several runtimes expose both fields.
Model profile modelProfile Selects simple, default, advanced, or advanced-long capability without naming a provider model. Adapter mappings resolve the native model and reasoning settings.
Stage model profiles modelStages Declares intended semantic profiles for bounded stages. Separate stage invocations must be implemented and captured before staged execution is considered verified.
Dynamic folder skills dynamicFolderSkills Requires the role to read the nearest AGENTS.md and load its declared folder technology skills before acting without rerunning detection.
Max turns maxTurns Caps the number of agentic turns or steps before the agent must return.
Timeout timeout Caps elapsed execution time or startup wait time. Keep this separate from max turns because timeouts are wall-clock constraints.
Permission mode permissionMode Controls when actions run automatically, ask for approval, or are blocked. This is not the same as tools because a visible tool or command may still require an approval decision.
Isolation isolation Controls filesystem, network, or workspace isolation where supported. Keep it separate from permission mode because sandboxing changes what the agent can reach, not only what it may ask to do.
Memory memory Controls persistent memory scope or names the guidance sources the agent should read before working.
Output contract outputContract Describes what the agent returns, what evidence it includes, and how another agent or human should continue from its result.

Codex

Native Definition

Codex custom agents are standalone TOML files under project .codex/agents/ or user ~/.codex/agents/. Each file defines one custom agent type and is loaded as a configuration layer for spawned sessions.

Identity And Prompt

The documented required fields are name, description, and developer_instructions. Codex identifies the agent from the name field rather than the filename.

Skills, Tools, And MCP

Codex subagents use the tools available to the parent task. Fixed-role skills are written as unconditional loading requirements in developer_instructions. Request-specific skills are written as conditional Use the skill instructions, including ambiguity guidance, and are not emitted as [[skills.config]]. Use [mcp_servers.name] for agent-specific MCP servers and emit [[skills.config]] only when the canonical role explicitly requests a name-based or path-based availability override; availability configuration is not preloading or read proof.

Runtime Controls

Because custom agents are config layers, the common source can map model, model_reasoning_effort, sandbox_mode, mcp_servers, and explicitly requested skills.config availability overrides into native TOML. Fixed-role skills alone do not create redundant config entries. Global controls such as agents.max_threads, agents.max_depth, and agents.job_max_runtime_seconds remain under [agents] in normal Codex config.

Config Files

Codex also reads user config from ~/.codex/config.toml, project overrides from trusted .codex/config.toml files, profile files such as ~/.codex/deep-review.config.toml, and one-off --config overrides. These layers are Codex implementation surfaces for shared concepts such as model, reasoning, permission mode, sandbox, MCP servers, hooks, and trust scope.

Rules Files

Codex rules files live under rules/ folders next to active config layers, such as ~/.codex/rules/default.rules or project-local .codex/rules/*.rules. Rules use prefix_rule entries to allow, prompt, or forbid matching commands outside the sandbox, so they map to the common permission-mode concept rather than becoming a separate role field.

Claude Code

Native Definition

Claude Code subagents are Markdown files with YAML frontmatter under .claude/agents/ for project scope or ~/.claude/agents/ for user scope.

Identity And Prompt

Only name and description are required. The Markdown body becomes the subagent system prompt.

Skills, Tools, And MCP

Claude Code documents a skills frontmatter field that preloads listed skills into the subagent context at startup. The generator maps only fixed role skills into that field. Request-specific skills stay out of frontmatter and appear as conditional loading instructions in the Markdown body.

Runtime Controls

Claude Code frontmatter can carry tool allowlists, model choice, permission or isolation behavior, MCP server configuration, hook behavior, memory handling, and turn limits. Unsupported common fields should remain in the body as explicit role constraints.

Gemini CLI

Native Definition

Gemini CLI custom agents are Markdown files with YAML frontmatter under .gemini/agents/ or ~/.gemini/agents/. The Markdown body becomes the agent system prompt.

Identity And Prompt

The documented configuration schema requires name and description, with optional kind, tools, mcpServers, model, temperature, max_turns, and timeout_mins.

Skills, Tools, And MCP

Gemini CLI documents Agent Skills and shared skill discovery tiers, but the current subagent schema does not document a skills frontmatter field. Fixed-role and folder skill-loading requirements should be emitted into the prompt body and backed by shared skill installation.

Runtime Controls

Gemini CLI maps the common source cleanly for tools, MCP servers, model, maximum turns, and timeout minutes. Adapter-specific controls such as temperature should be generated only when the Gemini adapter needs them. Permission, sandbox, memory, hook, and output rules that do not have native fields should stay in the body and in project guidelines.

Junie CLI

Native Definition

Junie CLI custom subagents are Markdown files with YAML metadata under project .junie/agents/, project .agents/, and matching user-scope locations.

Identity And Prompt

description is required. name is optional because Junie can infer it from the filename, but shared generation should emit explicit names for stability.

Skills, Tools, And MCP

Junie documents a skills field containing agent skill IDs or names for the subagent worker, when the environment supports it. The common role source can map role skill IDs directly to this field.

Runtime Controls

Junie metadata can carry tool and disallowed-tool fields, MCP server selection, model and reasoning preferences, maximum turns, and prompt-argument handling. Generation should keep the Markdown body as the authority for role behavior and output expectations.

GitHub Copilot Agent Mode

Native Definition

GitHub Copilot custom agents use Markdown files with a .agent.md extension. VS Code discovers workspace agents under .github/agents, supports Claude-format .claude/agents, and uses ~/.copilot/agents for user profile agents. Copilot cloud agent creation also starts from .github/agents/*.agent.md for repository agents.

Identity And Prompt

Copilot custom agent profiles use YAML frontmatter plus a Markdown prompt body. Documented fields include name, description, tools, agents, model, mcp-servers, target, and visibility or invocation controls. The profile body defines the agent behavior, expertise, and instructions.

Skills, Tools, And MCP

GitHub Copilot supports Agent Skills as SKILL.md folders and documents that those skills work with Copilot cloud agent, code review, CLI, the Copilot app, and agent mode in VS Code. The custom agent profile schema does not document a role-level skills frontmatter field, so role skill IDs should be emitted into the prompt body and backed by generated or installed Copilot skill folders.

Runtime Controls

Copilot profiles can carry tools, subagent references, MCP server configuration, model selection, target surface, invocation controls, and visibility metadata. Turn limits and output contracts should stay in the prompt body unless the target Copilot surface documents a native field.

Decision

Use A Common Role Source

Canonical files under agents/roles/ own name, filename, description, instructions, generic fixed-role skills, request-specific skill conditions, dynamic folder-skill policy, optional skill availability overrides, tools, disallowed tools, MCP servers, modelProfile, modelStages, max turns, timeout, permission mode, isolation, memory, and output contract. The generator validates them against agents/role-schema.yaml.

Generate Runtime Adapters

scripts/build-skill-docs.py resolves agents/model-profiles.yaml through each adapter model mapping, then emits Codex TOML, Claude Code Markdown, and design/generated/role-definitions.js. Gemini CLI, Junie CLI, and GitHub Copilot adapters follow the same source model in later adapter milestones.

Validate Against Skills

The generators build the skill catalog and setup-time technology detection registry before roles. Every canonical skills entry must resolve to a bundled generic skill; specialized variants are recorded in project AGENTS-PLAN.yaml and AGENTS.md instead of canonical roles.

Keep HTML Pure

agent-role-skill-map.html renders role cards, loadouts, outputs, and the skill inventory from generated data rather than defining a second catalog inline.

Fixed Roles And Setup-Time Technology Detection

Canonical role files own responsibility, authority, generic fixed-role skills, request-specific skill conditions, and output contracts. Project Agent Setup uses detect-technology-skills once for representative folders, while each specialized technology or domain skill owns its activation evidence in detection.yaml.

  • Project setup writes source-backed folder loadouts into AGENTS-PLAN.yaml and unconditional loading instructions into AGENTS.md.
  • Task execution loads fixed-role skills, conditionally loads request-specific skills after interpreting the request, and loads the nearest AGENTS.md folder skills without rerunning detection.
  • The detection build rejects specialized skills fixed directly into canonical role loadouts and rejects generic skills in detection metadata.
  • Claude preloads only fixed-role skills natively; Codex receives unconditional fixed-skill instructions. Both receive generated request-condition instructions. Harness traces remain necessary to prove runtime reads.
