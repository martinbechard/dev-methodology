Generic Agent Definitions Source

Skills and agents have different portability models. A source skill remains the same portable skill package when it is deployed, while a conceptual agent definition source must be converted into the file format and properties understood by each agent harness. This page explains that difference, the conceptual agent definition model, native agent packaging, cross-runtime property mappings, and vendor references.

Conceptual definition source Runtime adapters Skill validation Drift control

The Portability Problem

A repository that supports several agent harnesses must distinguish portable skill packages from harness-specific native agent definitions. Treating both as though they had the same source and runtime format creates duplicate agent files, ambiguous ownership, and drift.

Skills Have A Portable File Standard

The source skill definition is also the runtime skill definition. skills/<skill-name>/SKILL.md is copied into the selected skill directory without translating its frontmatter or instruction body into another format.

  • The portable SKILL.md remains authoritative before and after deployment.
  • Supporting files in the skill directory remain part of the same portable package.
  • Adjacent harness metadata, such as Codex agents/openai.yaml, supplements the package for that harness; it does not replace or rewrite SKILL.md.
  • A harness-specific skill source under adapters/<harness-name>/skills/ uses the same package format but is installed only with that adapter.

Agents Do Not Have One Portable Runtime File

Agent harnesses discover different files, use different serialization formats, and expose different property names and capabilities. The same conceptual agent definition might otherwise need separate hand-maintained TOML, Markdown, YAML frontmatter, and harness-specific configuration.

  • Discovery directories and filename extensions differ by harness.
  • Equivalent behavior may be a native property in one harness and instruction text in another.
  • Model, tool, permission, isolation, memory, and skill fields are not uniformly named or supported.

From Conceptual Agent Definitions To Native Agent Definitions

agents/roles/<role-category>/<role-name>.role.yaml is a conceptual agent definition source, not a file that an agent harness loads directly. It describes the agent once with the repository's shared vocabulary. The generator validates that vocabulary, resolves adapter-specific values, and emits one native agent definition per supported harness.

1. Author The Conceptual Agent Definition

Define intent with conceptual properties such as instructions, skills, modelProfile, tools, and outputContract. These properties describe the agent without making a harness file authoritative.

2. Validate And Resolve

agents/role-schema.yaml defines the allowed conceptual agent definition properties. Skill IDs resolve against portable skill sources, semantic model profiles resolve through adapter model mappings, adapter-owned skills resolve only within their harness, and unsupported or defaulted behavior follows the documented adapter rules.

3. Render Each Harness Format

scripts/build-skill-docs.py converts the conceptual values into native property names and file structures. For example, conceptual instructions becomes Codex developer_instructions or the Markdown body of a Claude Code, Gemini CLI, or Junie CLI agent. The Codex adapter also enables and adds a loading directive for codex-harness-directives when repositoryMutation is required or conditional.

4. Use Generated Native Agent Definition Files

The generated Codex TOML and Claude Code, Gemini CLI, and Junie CLI Markdown files are deployable runtime artifacts. They are regenerated from the conceptual agent definition source and must not become independent sources of truth.

Property conversion is not just a filename change

  • Rename or relocate: a conceptual property may become a differently named native property or move into an instruction body.
  • Resolve: a provider-neutral value such as modelProfile: advanced becomes the concrete model and reasoning settings selected by an adapter.
  • Expand: skill conditions, output contracts, and rationale can become native fields, generated instructions, or trace comments.
  • Default or omit: behavior without a native equivalent follows an explicit fallback rule instead of inventing an unsupported harness property.

Conceptual Agent Definition Properties

These are conceptual source-model properties, not a claim that every harness has a native field with the same name. The conceptual set captures the intent needed to define a reusable agent once. The later mapping table shows how each property becomes native metadata, instruction text, comments, a resolved value, or an explicit omission.

Element Conceptual Definition Property Meaning In The Source Definition
Name name Names the agent type used by routing, references, generated files, and UI display. Use a stable, short name that can survive adapter generation.
Filename filename Controls discovery and precedence in runtimes that load agent files from scoped folders. Keep it aligned with the name unless the runtime explicitly allows a different display name.
Description description Explains the agent's purpose and when a parent agent or user should delegate to it. This is a routing field, so it should be concise and specific.
Instructions instructions Defines behavior, boundaries, workflow, deliverables, and handoff expectations. Concise single-phase conceptual agent definitions may use a string. Definitions with branches, delegation, review loops, failure handling, or several terminal criteria use the validated structured mapping; generators render its sections into one portable runtime instruction string.
Skills skills Stores ordered skill entries with a skill identifier, justification, and optional condition. The skills modularization page explains when each entry applies.
Agent dependencies agentDependencies Names fixed direct agent-to-agent use declared by this conceptual definition. The generated diagram renders an arrow from the using agent to each used agent; dynamic task-time routing remains outside this list.
Skill condition skills[].condition Stores the optional human-readable condition beneath one skill entry; generators serialize it into runtime instructions.
Skill availability skillAvailability Stores explicit name-based or path-based availability overrides for adapters that expose them.
Tools tools Lists tools or tool groups the agent may use. Some runtimes treat this as an allowlist; some default to all tools when omitted.
Disallowed tools disallowedTools Lists tools or tool groups the agent must not use. Keep this separate from tools because allowlists and denylists resolve differently across runtimes.
MCP servers mcpServers Names or configures MCP servers available to the agent. Keep server access separate from tool names because several runtimes expose both fields.
Model profile modelProfile Selects simple, default, advanced, or advanced-long capability without naming a provider model. Adapter mappings resolve the native model and reasoning settings.
Stage model profiles modelStages Declares intended semantic profiles for bounded stages. Separate stage invocations must be implemented and captured before staged execution is considered verified.
Dynamic folder skills dynamicFolderSkills Records that the agent consumes folder skills supplied through project guidance; selection remains outside the conceptual definition source.
Repository mutation repositoryMutation Declares whether repository mutation is required, conditional, or never allowed for the agent. Generation validates that agent-claim is definition-owned, conditional, or absent to match. The Codex adapter separately uses required or conditional mutation to load its harness directives skill; this does not make those directives part of claim handling.
Max turns maxTurns Caps the number of agentic turns or steps before the agent must return.
Timeout timeout Caps agent-level elapsed execution time. An adapter must not substitute an unrelated MCP startup timeout.
Permission mode permissionMode Controls when actions run automatically, ask for approval, or are blocked. This is not the same as tools because a visible tool or command may still require an approval decision.
Isolation isolation Controls filesystem, network, or workspace isolation where supported. Keep it separate from permission mode because sandboxing changes what the agent can reach, not only what it may ask to do.
Memory memory Controls persistent memory scope or names the guidance sources the agent should read before working.
Output contract outputContract Describes what the agent returns, what evidence it includes, and how another agent or human should continue from its result.
Examples examples Stores customer-safe runtime invocations and plausible responses for generated catalog views and regression checks.
