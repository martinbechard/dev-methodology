Agentic Development Operating Model

This page owns project classification, guidance precedence, lifecycle handoffs, and execution evidence. Use the generated catalog for current role and skill definitions, the specialization strategy for selection policy, and runtime formats for adapter details.

Project Classification

PROJECT.yaml records the project classifications that affect routing. These values describe the target project; they do not duplicate the generated role or skill catalogs.

Dimension Question it answers Routing effect
Project family What product or repository boundary is being maintained? Establishes the relevant tiers, documentation surfaces, and default validation concerns.
Application tier Which local responsibility and stack boundary owns the affected files? Maps folder scopes to technology skills, local guidance, and checks.
Artifact type What durable output is requested? Selects the creation, review, and placement route without creating a framework-specific role.
Data boundary Which private, proprietary, generated, or regulated information is in scope? Adds handling constraints, approvals, and local evidence requirements.
Validation surface What evidence proves the work is usable? Names the relevant build, test, documentation, runtime, or review checks.

Common Web Application Tiers

Tier Responsibility boundary
Front-end Interactive screens, components, client and server rendering boundaries, styling, accessibility, and browser behavior.
API Routes, controllers, services, authentication, request validation, contracts, and integration behavior.
Static pages Marketing, documentation, legal, and other content routes with static rendering, accessibility, SEO, and visual checks.
DB Schema, migrations, queries, persistence, seed data, and database integration behavior.

Other project families declare only the tiers they actually contain. Concrete root-only, nested, and wiki configurations appear on the examples page.

Project Guidance And Precedence

PROJECT.yaml is the project-root setup and validation record. It explains role selection, folder loadouts, nested-guidance placement, and supporting evidence; ordinary work consumes the resulting guidance rather than treating the configuration as another instruction layer.

Order Source Responsibility Boundary
1 Host rules Runtime safety, tool permissions, privacy constraints, and sandbox behavior. Lower layers cannot disable them.
2 Selected role contract Purpose, authority, expected inputs, output contract, and handoff behavior. Project guidance cannot change role authority or verdict shape.
3 Root AGENTS.md Repository-wide rules, folder loadouts, validation commands, data boundaries, and local reference paths. It selects shared guidance without copying complete role or skill definitions.
4 Nearest nested AGENTS.md Only the stack, validation, generated-file, data, or workflow differences for its subtree. It narrows routing but cannot bypass safety, review evidence, or project-wide constraints.
5 Selected skills Fixed, request-specific, and setup-established folder expertise selected by the specialization policy. The specialization strategy is the sole owner of those selection rules.
6 Task request and evidence The requested outcome, target files, constraints, and acceptance evidence. The task can narrow work but cannot override higher-priority constraints.

For Claude Code, a thin CLAUDE.md may import the colocated AGENTS.md. It does not duplicate the project guidance.

Orchestrated Development Loop

Step Owner Transition or gate
Bootstrap, when requested Project Bootstrapper Reuse a valid project-root PROJECT.yaml. When it is absent, or when the user explicitly requests reconfiguration, Project Configurator must create and validate it before documentation work begins.
Intake Dev Orchestrator Establish the task boundary, assignments, and validation route before handing work to the selected roles.
Claim gate Every modifying claim owner Mutation begins only after agent-claim establishes ownership of the affected files and exclusive resources.
Placement, when needed Project Organiser A new artifact proceeds only after its repository path is approved.
Execution Selected implementation or writing role The assigned output passes to an independent reviewer; the generated catalog owns each role's detailed inputs, skills, and output contract.
Review Selected independent review role Correctable findings return to the owning executor; accepted work advances to integrated verification.
Verification Dev Verifier The configured acceptance checks must pass, or an unresolved risk must be recorded, before integration.
Integration Dev Merge Coordinator Only committed, clean contributions with verification evidence enter the integration lane.
Completion Claim owner Release ownership only after verification, commit or explicit no-change, clean status, and resource cleanup or handoff.

Execution Evidence

An orchestrated run leaves four complementary records. Each fact has one record owner so later readers do not have to reconcile competing summaries.

Record Owns Does not own
.agents/runs/<run-id>/context-pack.json Inputs supplied to the assigned role: files, applicable guidance, constraints, and selected skills. Results or verification claims.
.agents/runs/<run-id>/agent-results.jsonl Ordered role outputs and status events for replay, audit, and merge coordination. A synthesized verification verdict.
.agents/runs/<run-id>/verification.md Checks run, outcomes, unresolved risks, and justified omissions. Task history or next-owner context.
.agents/runs/<run-id>/handoff.md Completed work, remaining work, follow-up ownership, and the context needed to continue. Raw event history or duplicated command transcripts.
