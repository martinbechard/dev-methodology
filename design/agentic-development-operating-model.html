Agentic Development Operating Model

The methodology uses named role agents for human and orchestrated work, short skills for load-on-demand procedures, project profiles for selecting stack and tier guidance, and folder policies for local overrides. Each layer has one job, so long-running development can split into subagents without making every context carry every rule.

Operating Layers

1. Project Profile

Declares project type, tier, stack packs, enabled agents, validation commands, data rules, and folder policy roots.

2. Role Agent

Defines an invokable job role with a purpose, triggers, allowed tools, expected inputs, skill loadout, outputs, and handoff rules.

3. Skill

Provides a short procedural instruction unit that a role loads only when the job requires that expertise.

4. Orchestrator

Breaks work into bounded subagent jobs, prepares context packs, routes outputs, and runs review or retry loops.

5. Runtime Adapter

Materializes shared role definitions into host-specific agent files for Codex, Claude Code, Gemini CLI, Junie, or another runtime.

Skill Load Model

Skill contract

Skills normalize reusable information into short definitions. A skill contains the narrow procedure, trigger, constraints, and assets for one expertise area.

  • A skill is not an agent persona.
  • A role agent loads the skill only when the task needs it.
  • A skill stays portable across projects and runtimes.
  • Project-specific selection lives in the project profile.

Skill groups

The profile selects skill groups by project type, stack, tier, and folder scope.

  • Methodology skills for docs, wiki, structured design, review, backlog, and merge work.
  • Stack skills for Next.js, Electron, TypeScript, Postgres, LangGraph, local model integration, and browser tests.
  • Quality skills for tests, security, privacy, performance, observability, and release checks.
  • Repository skills for local procedures, taxonomy, scripts, templates, and generated artifact conventions.

Project Tiers

Prototype or spike

Fast learning, narrow scope, explicit throwaway boundaries, lightweight tests for important assumptions.

Greenfield product

Simple steady-state design, active refactoring, full build verification, minimal migration ceremony.

Internal tool

Practical reliability, clear ownership, repeatable setup, focused docs, and efficient verification.

Production service

Build and test gates, observability, security checks, rollback clarity, and operational runbooks.

Regulated or high-risk system

Traceability, privacy controls, audit evidence, threat review, and explicit approval records.

Legacy maintenance

Compatibility constraints, conservative edits, migration plans, fixture coverage, and regression review.

Project Profile

Root profile

The root profile is the normal selector for project family, tier, stack packs, role availability, validation commands, and data handling rules.

  • Canonical path: .agents/project-profile.yaml
  • Host adapters may mirror selected fields into runtime-specific config files.
  • The profile selects skills and policies; it does not duplicate full skill bodies or role prompts.

Folder profile

Folder profiles narrow or add guidance for subtrees with a different stack, test policy, artifact type, or data sensitivity.

  • Canonical path: .agents/folder-profile.yaml inside the governed subtree.
  • Nearest folder profile wins for files inside that subtree.
  • Folder profiles contain only the fields that differ from the root profile.
project:
  name: example-product
  family: full-stack-product
  tier: greenfield-product
  primary_stack: nextjs-app-router
  secondary_stacks:
    - postgres-drizzle
    - langgraph

  default_orchestrator: development-orchestrator
  enabled_roles:
    - project-organiser
    - coding-agent
    - code-review-agent
    - qa-and-verification-agent
    - documentation-architect

skills:
  methodology:
    - careful-coding
    - structured-review
    - fix-explanation
  stack_packs:
    - nextjs-app-router
    - react-server-components
    - typescript-strict
    - api-routes
    - clerk-auth
    - postgres-drizzle

Guidance Resolution

Host rules

Runtime safety rules, tool permissions, and privacy constraints are always active.

Role contract

The selected agent role supplies purpose, input requirements, allowed tools, output shape, and handoff behavior.

Root profile

The project profile selects project family, tier, stacks, skill packs, and validation commands.

Folder profile

The nearest folder profile supplies local stack or policy overrides for the touched files.

Task evidence

The task prompt and repository evidence select the smallest relevant subset of skills for the job.

Hard safety, privacy, and data handling rules remain constraints at every layer. Task prompts can narrow work, but they do not disable higher-priority constraints.

Orchestrated Development Loop

Step Owner Context boundary Output
Intake Development Orchestrator Reads the user request, project profile, relevant folder profiles, backlog, and repo status. Task breakdown, agent assignments, context pack list, validation plan.
Placement Project Organiser Reads taxonomy, requested artifact descriptions, and path intents. Approved paths, taxonomy updates when needed, placement rationale.
Execution Coding Agent or Documentation Architect Receives only the files, skills, profile excerpts, and acceptance criteria needed for the assigned job. Patch, artifact, test evidence, build evidence, implementation notes.
Review Code Review Agent or Artifact Review Agent Receives the diff, source evidence, role contract, and profile rules without the executor's full scratch context. PASS WARN FAIL verdict with file and rule references.
Verification QA And Verification Agent Receives the acceptance criteria, validation commands, changed files, and relevant runtime notes. Command results, coverage notes, unresolved risks.
Integration Merge Coordinator Receives branch state, claims, conflicts, and verification records for parallel work. Merged work, conflict ownership, remaining follow-up items.
