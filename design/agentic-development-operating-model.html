Agentic Development Operating Model

The methodology uses named role agents for human and orchestrated work, short skills for load-on-demand procedures, AGENTS.md files for operational project-family, application-tier, folder-skill, validation, and data-handling reference. Each layer has one job, so long-running development can split into subagents without making every context carry every rule.

Operating Layers

1. AGENTS.md Reference

Declares the project family, application tiers, unconditional folder technology loadouts, validation commands, data rules, and local reference information.

2. Role Agent

Defines an invokable job role with a purpose, triggers, allowed tools, expected inputs, skill loadout, outputs, and handoff rules.

3. Skill

Provides a short procedural instruction unit that a role loads only when the job requires that expertise.

4. Orchestrator

Breaks work into bounded subagent jobs, prepares context packs, routes outputs, and runs review or retry loops.

5. Runtime Adapter

Materializes shared role definitions into host-specific agent files for Codex, Claude Code, Gemini CLI, Junie, or another runtime.

Role Agent Set

Wiki setup, query, research, collection, writing, ingest, topic verification, and project-wiki artifact review form a dedicated activity category. General documentation, coding, review, backlog, and project-setup roles do not load wiki skills.

Skill Load Model

Skill contract

Skills normalize reusable information into short definitions. A skill contains the narrow procedure, trigger, constraints, and assets for one expertise area.

  • A skill is not an agent persona.
  • A role agent loads the skill only when the task needs it.
  • A skill stays portable across projects and runtimes.
  • Project-specific selection lives in AGENTS.md files.
  • Detailed code-generation rules and review-evidence questions live together in language, framework, tier, persistence, and test skills.

Skill groups

Project Agent Setup detects specialized guidance once from generated activation metadata and representative folder evidence. AGENTS.md then tells ordinary agents which folder skills to load before acting.

  • Methodology skills for docs, wiki, artifact creation, structured design, review, backlog, and merge work.
  • Specialized skills for languages, frameworks, persistence, identity, runtimes, interface systems, tests, and optional tools.
  • Quality skills for tests, security, privacy, performance, observability, and release checks.
  • Repository skills for local procedures, taxonomy, scripts, templates, generated artifact conventions, and deterministic path bindings.
  • Detection results record setup evidence and availability; runtime harness evidence separately proves whether a skill file was read.

Application Tiers

Front-end tier

User-facing application screens, interactive components, client/server rendering boundaries, styling, accessibility, and browser workflow guidance.

API tier

Route handlers, server actions, controllers, services, authentication boundaries, request validation, contracts, and integration test guidance.

Static pages tier

Marketing, documentation, legal, and static content routes with content structure, SEO, accessibility, static rendering, and visual verification guidance.

DB tier

Schema, migrations, queries, persistence modules, seed data, data sensitivity rules, repository patterns, and database integration test guidance.

A web application declares these tiers by default. Other project families declare only the tiers they actually contain, then map each tier to the relevant stack packs, nested AGENTS.md guidance, and validation commands.

AGENTS.md Operational Reference

Root AGENTS.md

The root AGENTS.md is the normal source for project family, application tiers, folder technology loadouts, validation commands, and data handling rules.

  • Canonical path: AGENTS.md
  • It tells every agent which folder technology skills to load before acting.
  • It selects skills and policies; it does not duplicate full skill bodies or role prompts.

Nested AGENTS.md

Nested AGENTS.md files narrow or add guidance for subtrees with a different application tier, stack, test policy, artifact type, or data sensitivity.

  • Canonical path: AGENTS.md inside the governed subtree.
  • The nearest nested AGENTS.md supplies local router reference for files inside that subtree.
  • Nested files contain only the fields and rules that differ from the root AGENTS.md.

Guidance Resolution

Host rules

Runtime safety rules, tool permissions, and privacy constraints are always active.

Role contract

The selected agent role supplies purpose, input requirements, allowed tools, output shape, and handoff behavior.

Root AGENTS.md

The root AGENTS.md names the project family, application tiers, folder technology loadouts, and validation commands.

Nested AGENTS.md

The nearest nested AGENTS.md supplies local stack, tier, validation, or policy reference for the touched files.

Task evidence

The role always loads its fixed skills. The agent interprets the requested outcome and evidence to apply request-specific skill conditions, including ambiguous wording, while the folder determines which pre-established AGENTS.md technology loadout applies.

Hard safety, privacy, and data handling rules remain constraints at every layer. Task prompts can narrow work, but they do not disable higher-priority constraints.

File Contracts

File contracts make the operating model concrete. They tell agents where durable role definitions live, where project-local operational guidance lives, which files adapt shared definitions to a runtime, and where completed runs leave evidence. A small project can start with AGENTS.md and add the other contracts as orchestration becomes useful.

Shared role source

These files define reusable role behavior before it is rendered into Codex, Claude Code, or another host.

  • agents/roles/development-use/development-orchestrator.role.yaml defines the coordinator role that breaks work into bounded tasks, selects agents, and preserves handoff and verification expectations.
  • agents/roles/project-setup/project-organiser.role.yaml defines the placement role that applies taxonomy, path intent, and file-organization rules before artifacts or code are created.
  • agents/roles/development-use/coding-agent.role.yaml defines the implementation role used for ordinary code changes after fixed and folder skills are loaded.
  • agents/roles/development-use/code-review-agent.role.yaml defines the review role, including finding-first output, severity expectations, evidence requirements, and verification checks.
  • agents/roles/wiki-activities/*.role.yaml defines the separate wiki setup, query, research, collection, writing, ingest, verification, and project-wiki artifact review responsibilities.

Operational reference

These files provide project-specific facts and unconditional folder skill loadouts.

  • AGENTS.md is the root project contract. It declares the project family, application tiers, validation commands, data rules, and default guidance for this repository.
  • AGENTS-PLAN.yaml is the setup and validation artifact that explains why project role agents, folder technology loadouts, nested plans, and AGENTS.md references exist.
  • <subtree>/AGENTS.md narrows the root contract for a folder that has different tier, stack, validation, or data-handling needs. The nearest file applies to work in that subtree.
  • CLAUDE.md is a thin Claude Code bridge that imports the colocated AGENTS.md; create the same bridge beside nested guidance when Claude needs that subtree.
  • <subtree>/AGENTS-PLAN.yaml is used only when the subtree has distinct technology, runtime, data, or verification evidence that would make the root plan too noisy.
  • docs/project-taxonomy.md defines the project families, artifact types, and placement concepts used by Project Agent Setup and Project Organiser.

Runtime adapters

These files translate shared roles and skills into the format a specific agent runtime actually consumes.

  • generated/adapters/codex/agents/*.toml contains generated Codex custom-agent definitions ready for installation under .codex/agents/ or ~/.codex/agents/.
  • generated/adapters/claude/agents/*.md contains generated Claude Code definitions ready for installation under .claude/agents/ or ~/.claude/agents/.
  • skills/*/agents/openai.yaml stores optional Codex metadata beside the source skill so deployment matches the Codex skill layout while sync tooling keeps derived interface fields aligned.

Generated documentation

The HTML pages are views over canonical skill and role catalogs. The maintain-methodology-documentation skill updates sources, regenerates browser data and runtime adapters, then runs stale-output and consistency checks.

Customer installation and maintenance

Deterministic installers copy generic skills and native agents with ownership manifests. Customer edits keep stable names, are never silently overwritten, and are reconciled through three-way discrepancy analysis during an update.

Execution evidence

These files make an orchestrated run inspectable after the fact, so another agent or human can see what context was used and what was verified.

  • .agents/runs/<run-id>/context-pack.json captures the files, AGENTS.md excerpts, task constraints, and selected skills given to the assigned agent.
  • .agents/runs/<run-id>/agent-results.jsonl records agent outputs and status events in order, which supports replay, audit, and merge coordination.
  • .agents/runs/<run-id>/verification.md records the checks that were run, their results, unresolved risks, and any reason a normal check was skipped.
  • .agents/runs/<run-id>/handoff.md summarizes what changed, what remains, who owns follow-up work, and what context the next agent needs.
