Agentic Configuration

This page describes runtime use. It does not describe the source model that creates agent files. The Generic Agent Definitions Source page owns that source model and its conversion into harness-specific files.

Provide The Relevant Context

In order to generate usable code, we need to provide all of the relevant context that will steer the agent to generating it in a way that is acceptable to us. The main problem is choosing the relevant context from many possible units of information. The solution is to split up the information into many files and have the agentic coding tool load just the skills it needs.

Context Layers

  1. Shared Definitions

    Definitions installed for a harness are available to all projects that use that harness.

    1. Skill Definition Files

      The Agent Skills format uses a text file named SKILL.md to describe how to perform actions. It is adopted by all vendors and is the most granular unit of description.

    2. Agent Definition Files

      Agent definition files describe the purpose of a specific agent, the skills and other agents it should use, and its other directives. Each agent operates with its own context and history, which allows us to partition the amount of information being worked on at a given time by creating a hierarchy of agents.

  2. Project-Specific Definitions

    Definitions and instructions stored within a project apply only to that project and can tailor shared behavior to its content.

    1. Project Skill Definition Files

      Project skill definition files describe actions that are specific to the project or customize a shared skill for the project.

    2. Project Agent Definition Files

      Project agent definition files define project-specific agents or customize shared agent definitions for the project.

    3. Root Project Instruction Files

      Project-specific instructions can be defined using AGENTS.md and CLAUDE.md placed at the root of the project. This project guidance should usually act as a router to load skills appropriate for the project content.

    4. Nested Project Instruction Files

      More of these instruction files can be nested to provide specialized guidance for content within a project folder.

Runtime Configuration File Locations

The placeholders identify the harness, project, folder, skill, and agent names directly. These paths do not use filename globs.

File Shared Location Project Location Runtime Purpose
Skill definition ~/.<harness-name>/skills/<skill-name>/SKILL.md <project-root>/.<harness-name>/skills/<skill-name>/SKILL.md Describes how to perform actions.
Markdown agent definition ~/.<harness-name>/agents/<agent-name>.md <project-root>/.<harness-name>/agents/<agent-name>.md Defines a specific role to be performed by the AI using skills and other directives.
TOML agent definition ~/.<harness-name>/agents/<agent-name>.toml <project-root>/.<harness-name>/agents/<agent-name>.toml Defines a specific role to be performed by the AI using skills and other directives.
Root agent instructions Not shared <project-root>/AGENTS.md Defines project-specific instructions and acts as a router to load skills appropriate for the project content.
Root Claude instructions Not shared <project-root>/CLAUDE.md Defines project-specific instructions and acts as a router to load skills appropriate for the project content.
Nested agent instructions Not shared <project-root>/<folder-path>/AGENTS.md Provides even more specialized instructions for nested project content.
Nested Claude instructions Not shared <project-root>/<folder-path>/CLAUDE.md Provides even more specialized instructions for nested project content.

Actual harness directory names vary. Shared locations make agent and skill definitions available among projects; project locations place those definitions within a project.
