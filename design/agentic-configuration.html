Agentic Configuration

Agentic configuration is the set of runtime files an agentic coding tool loads to decide which instructions, roles, and skills apply while it generates code.

This page describes runtime use. It does not describe the source model that creates agent files. The Generic Agent Definitions Source page owns that source model and its conversion into harness-specific files.

Provide The Relevant Context

To generate usable code, the agent needs the relevant context that will steer it toward a result acceptable to the project. The main problem is selecting that context from many possible units of information. Agentic configuration solves this by splitting the information into focused files and having the coding tool load only the skills and instructions needed for the current work.

Project Instructions

Identify the guidance and skills that apply to the project or folder.

Agent Definition

Establish the role, its directives, its skills, and the agents it can use.

Relevant Skills

Supply the detailed instructions needed to perform the selected actions.

Generated Code

Reflects the project expectations contained in the loaded runtime context.

Skills And Agent Definitions

Skills Describe Actions

The Agent Skills format uses a text file named SKILL.md to describe how to perform actions. Agentic coding vendors have adopted this format as the most granular unit of reusable instruction. Keeping skills focused allows the tool to load the relevant action guidance without loading unrelated material.

Agent Definitions Describe Roles

An agent definition specifies a role for the AI to perform. The configuration is primarily a description of the agent's purpose, the skills it should use, other directives it must follow, and the other agents it may use to complete the work.

Agent Hierarchies Partition Context

Each agent operates with its own context and history. A parent agent can delegate focused work to other agents, creating a hierarchy that limits how much information and history any one agent works with at a time.

Shared, Project, And Nested Configuration

Agent and skill definitions are usually shared among projects, but they can also be stored inside a project when the role or action is project-specific.

Root Project Instructions

AGENTS.md and CLAUDE.md at the project root provide project-specific instructions. They should usually act as routers that direct the coding tool to the skills appropriate for the project's content.

Nested Project Instructions

Additional AGENTS.md and CLAUDE.md files can be placed in nested folders. These files provide more specialized instructions or skill routing for the content below that folder.

Runtime Configuration File Locations

The placeholders identify the harness, project, folder, skill, and agent names directly. These paths do not use filename globs.

File Shared Location Project Location Runtime Purpose
Skill definition ~/.<harness-name>/skills/<skill-name>/SKILL.md <project-root>/.<harness-name>/skills/<skill-name>/SKILL.md Provides focused instructions for performing one kind of action.
Markdown agent definition ~/.<harness-name>/agents/<agent-name>.md <project-root>/.<harness-name>/agents/<agent-name>.md Defines a role for a harness that loads Markdown agent files.
TOML agent definition ~/.<harness-name>/agents/<agent-name>.toml <project-root>/.<harness-name>/agents/<agent-name>.toml Defines a role for a harness that loads TOML agent files.
Root agent instructions Not shared <project-root>/AGENTS.md Provides project-wide instructions and routes work to relevant skills.
Root Claude instructions Not shared <project-root>/CLAUDE.md Provides the Claude-specific entry point for project-wide instructions.
Nested agent instructions Not shared <project-root>/<folder-path>/AGENTS.md Provides specialized instructions and skill routing for one folder subtree.
Nested Claude instructions Not shared <project-root>/<folder-path>/CLAUDE.md Provides the Claude-specific entry point for the nested folder instructions.

Actual harness directory names vary. Shared locations make a definition available across projects; project locations travel with one repository.
