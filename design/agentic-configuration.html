Agentic Configuration

This page describes runtime use.

Knowledge Structure

In order to generate usable code, we need to provide all of the relevant context that will steer the agent to generating it in a way that is acceptable to us. The main problem is choosing the relevant context from many possible units of information. The solution is to split up the information into many files and have the agentic coding tool load just the skills it needs.

Context Layers

  1. Shared Definitions

    Definitions installed for a harness are available to all projects that use that harness.

    1. Skill Definition Files

      The Agent Skills format uses a text file named SKILL.md to describe how to perform actions. It is adopted by all vendors and is the most granular unit of description. Adapter-owned skill definitions use the same SKILL.md format but are installed only for their matching harness.

    2. Agent Definition Files

      Harness-specific agent definition files describe the purpose of a specific agent, the skills and other agents it should use, and its other directives. Each agent operates with its own context and history, which allows us to partition the amount of information being worked on at a given time by creating a hierarchy of agents.

  2. Project-Specific Definitions

    Definitions and instructions stored within a project apply only to that project and can tailor shared behavior to its content.

    1. Project Skill Definition Files

      Project skill definition files describe actions that are specific to the project or customize a shared skill for the project.

    2. Project Agent Definition Files

      Project agent definition files define project-specific agents or customize shared harness-specific agent definitions for the project.

    3. Root Project Instruction Files

      Project setup creates a portable AGENTS.md at the project root. This guidance should usually act as a router to load skills appropriate for the project content. For Claude Code, a thin colocated CLAUDE.md imports that root guidance without duplicating it.

    4. Nested Project Instruction Files

      Project setup can place another portable AGENTS.md in a folder that needs specialized guidance. For Claude Code, a thin CLAUDE.md sits beside every applicable nested AGENTS.md and imports the colocated guidance.

Runtime Configuration File Locations

Each group starts with the context purpose, then shows the runtime locations and rules for each harness. Every harness uses the same portable Agent Skills package: a skill directory containing SKILL.md with YAML frontmatter and Markdown instructions. Skill locations and precedence vary; the SKILL.md format does not.

A Codex deployment includes the adapter-owned codex-harness-directives skill in addition to shared skills. Codex-generated agents that may mutate the repository enable it and instruct the agent to load it before acting. Read-only Codex agents and non-Codex adapters do not receive it. The source-model page owns the generation rule.

The preferred mcp-agent-ops connection is host or session configuration, separate from skill and agent deployment. Codex defines the stdio server in user or trusted-project config.toml; a Codex agent inherits parent servers when it omits mcp_servers, while an override must contain full runtime-specific server configuration. Junie defines the server in user or project mcp.json; a nonempty agent mcpServers field is a names-only allowlist, and omission uses the servers configured for the session. The bundle installer never writes either host configuration.

The cross-harness evaluation runner uses the same separation. It pins the mcp-agent-ops release and installed-runtime digest, stages an exact catalog beneath a disposable context root, supplies one isolated Codex or external Junie server configuration, and retains session-bound process streams plus exact configuration, catalog, authorization, and allowed output evidence outside the product workspace. The MCP Git-lifecycle case gives Codex an exact digest-bound permission profile and gives Junie an owner-only action policy for the same ten MCP operations plus Git add and Git commit. The Codex profile denies network, host-home, and evidence-root access while granting writes to the disposable workspace, its Git metadata, and the unique evaluator temporary directory; other system paths remain read-only. The Junie policy controls capabilities but is not filesystem containment. Each required operation is bound to its privacy-safe argument digest and bounded outcome on the same completed call. These boundaries protect reproducibility and host state; they do not claim hostile-code containment.

Nested Project Instruction FilesAdd or override instructions for a folder, path pattern, or narrower working scope.
Codex Inherited from ~/.codex/AGENTS.md <folder-path>/AGENTS.md or <folder-path>/AGENTS.override.md Codex walks only from the project root to the current working directory; guidance closer to that directory appears later and overrides broader guidance.
Claude Code Inherited through the applicable CLAUDE.md bridge <folder-path>/AGENTS.md with a colocated <folder-path>/CLAUDE.md The thin CLAUDE.md imports the colocated AGENTS.md so Claude uses the same portable folder guidance as other harnesses.
Gemini CLI Inherited from ~/.gemini/GEMINI.md <folder-path>/GEMINI.md Gemini scans ancestor and subdirectory context files, then concatenates all discovered instructions with their source paths.
Junie CLI Inherited from ~/.junie/AGENTS.md Not a documented nested guideline location Junie documents root project guidelines rather than nested discovery. Use project skills for task-specific or folder-specific procedures.
GitHub Copilot Inherited from ~/.copilot/copilot-instructions.md <project-root>/.github/instructions/<rule-name>.instructions.md or <folder-path>/AGENTS.md Path-specific instruction files require YAML applyTo frontmatter. Non-root AGENTS.md files are additional rather than primary instructions.

The bundle installer maps --scope user and --scope project to the matching locations in this table. Explicit skill and agent destination arguments override those defaults. Cleanup of obsolete bundle-owned artifacts is enabled by default and can be retained for a run with --cleanup false.

The Generic Agent Definitions Source page owns conceptual-to-harness-specific conversion details. This table covers only what each runtime file contributes, where the harness discovers it, and which incompatible harness-specific rule or format applies.
