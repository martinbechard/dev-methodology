Agent Roles and Skill Loadouts

The methodology should separate four things that are currently easy to blur: agent definition files describe a role, skills describe reusable procedures, project profiles select the pertinent best practices, and repo automations or prompt modes describe how a role is run in a specific environment.

Working Model

1. Role definition

A named agent such as Orchestrator Agent, Coding Agent, Code Review Agent, Wiki Ingest Agent, or Project Organiser.

2. Skill loadout

Short reusable procedures loaded only when the current job needs them, such as careful-coding, ast-grep, or project-wiki-topic-verifier.

3. Project profile

A root or folder-local selector for project type, tier, stack, conventions, enabled agents, and best-practice packs.

4. Runtime adapter

The concrete file format for a host, such as the current Codex openai.yaml metadata files.

5. Repo workflow

The local operating surface, such as raw-only collectors, leaf watches, Gemma prompt modes, worktree runs, or orchestrated subagent queues.

Proposed direction: define composite role agents once, then generate or copy runtime-specific definitions for Codex, Claude Code, and other hosts. Skills remain reusable and shared across roles, while project profiles decide which guidance is relevant for a given repo or folder.

Proposed Composite Agent Definitions

Development Orchestrator

Breaks long-running work into subagent jobs, gives each agent a bounded context pack, tracks handoffs, and decides when to review, retry, merge, or escalate.

Project Organiser

Classifies new artifacts by purpose, chooses paths from a living taxonomy, and checks that generated files landed in the right project folder.

Owns a scoped source change from repo orientation through patch, focused tests, build, and final explanation using the active stack guidance.

Project Type And Stack Taxonomy

Project families

Families answer what kind of work the repository mainly supports.

  • Frontend web app or design system.
  • API service or backend domain service.
  • Full-stack product with web, server, data, and deployment concerns.
  • CLI, local tooling, or developer productivity utility.
  • Data, analytics, ingestion, or reporting pipeline.
  • AI agent harness, eval runner, prompt system, or orchestration framework.
  • Documentation, wiki, research, or curriculum repository.
  • Library, package, SDK, or reusable component set.
  • Enterprise integration, workflow, or systems automation project.

Project tiers

Tiers answer how much rigor the role should apply before changing or approving work.

  • Prototype or spike: fast learning, low ceremony, explicit throwaway boundaries.
  • Greenfield product: simple steady-state design, strong refactoring, no legacy fallbacks.
  • Internal tool: practical reliability, clear ownership, lean documentation.
  • Production service: full build and test gates, monitoring, security, rollback clarity.
  • Regulated or high-risk system: traceability, audit evidence, privacy and security reviews.
  • Legacy maintenance: compatibility constraints, migration plans, conservative change scope.

Stack packs

Stack packs select concrete best practices without baking them into every agent prompt.

  • Next.js full-stack: nextjs-app-router, react-server-components, typescript-strict, api-routes, clerk-auth, postgres-drizzle, langgraph, tailwind-design-system, jest, and playwright.
  • Electron local model app: typescript-esm, electron-main, electron-preload, react-vite-renderer, node-cli, vitest, and local-model-integration.
  • Custom harness domain: harness-implementation, tool-runtime-implementation, plan-engine-implementation, and local-model-integration.
  • Wiki or documentation repo: source separation, durable leaves, raw ingest, link validation, topic verification.
  • Additional languages and frameworks can add packs through the same catalog shape.

Runtime Definition Strategy

Current Codex adapter

Keep the existing skill metadata files. They are useful launch shortcuts, but they should not be mistaken for full role definitions.

  • Current shape: adapters/codex/skills/*/agents/openai.yaml
  • Current behavior: one skill gets one display name and one default prompt.
  • Gap: no role-level file that references a multi-skill loadout.

Proposed role source

Add a source-of-truth folder for composite role definitions, then adapt it to host-specific locations.

  • Candidate source folder: agents/roles
  • Candidate fields: role, purpose, trigger, inputs, allowed tools, skills, outputs, verification, handoff.
  • Installer change: copy or generate runtime files for Codex and Claude Code from the shared role source.

Skill boundary

A skill should stay small and procedural. A role can combine skills, decide when to use each one, and define the final output contract.

  • One role can use many skills.
  • One skill can serve many roles.
  • Automations can bind a role to a schedule or queue.

Subagent boundary

For long-running development, the orchestrator should give each subagent a narrow brief and its own context, then reconcile outputs through review and verification.

  • Interactive use: a human invokes a named role directly.
  • Autonomous use: the orchestrator invokes roles as bounded work units.
  • Review use: fresh-context agents inspect outputs without inheriting implementation assumptions.

Project Profile And Folder Overrides

Minimal config pattern

Use one root profile as the normal selector, then allow folder-local overrides only when a subtree really follows a different stack or policy.

  • Root profile candidate: .agents/project-profile.yaml
  • Folder override candidate: .agents/folder-profile.yaml in the subtree it governs.
  • Folder overrides should add or narrow guidance, not duplicate the whole root profile.
  • The profile should select skills; it should not become a second agent prompt language.

Resolution order

The runtime should resolve guidance from broad to narrow, with explicit safety and privacy rules treated as hard constraints.

  • Host defaults and global safety rules.
  • Methodology bundle and role definition.
  • Root project profile.
  • Nearest folder profile for the files being touched.
  • Task prompt and source evidence.
