Agent Roles and Skill Loadouts

This reference defines the portable agent roles and skill loadouts in the development methodology bundle. It is customer-ready: roles are generic, project-specific setup is documented in AGENTS-PLAN.yaml, operational routing comes from AGENTS.md files, and skills stay small enough to load only when the current job needs them.

Reference Model

1. Role

A role is an invokable job boundary with stable authority, inputs, outputs, and handoff expectations.

2. Router Reference

Root and nested AGENTS.md files describe project family, application tiers, paths, checks, data rules, and local terminology.

3. Skill Selection

The role loads only the methodology, stack, quality, wiki, or domain skills needed for the current request.

4. Runtime Adapter

Codex, Claude Code, Gemini CLI, Junie, or another host receives a runtime-specific role file or skill metadata.

5. Evidence

Review checklists, verification notes, handoffs, and run records make the work inspectable after completion.

AGENTS.md is the project-local operational reference. It should bind technology skills to the right folders and name validation and data rules without copying shared skill bodies or redefining role authority.

Role Agent Categories

The role model has four operating categories: agents that maintain this methodology bundle, agents that set up or update a downstream project, agents that own wiki activities, and agents used during normal development in that downstream project.

Amber skills are fixed role skills. Violet skills are request-specific and are loaded only when their condition applies. Blue skills in the catalog are setup-time technology skills; Project Agent Setup binds them to folders through AGENTS.md rather than assigning them to one canonical role.

Routing Rules

Signal Router action Example loadout
User changes this methodology bundle Use a Methodology Maintenance Agent so source skills, metadata, README, design HTML, tests, and shared-install behavior stay aligned. development-methodology, review-structured, documentation-page-verifier, validation scripts, and metadata scripts as needed.
User sets up or updates a downstream project Use a Project Setup And Update Agent to create or refresh AGENTS-PLAN.yaml, local AGENTS.md routing, documentation routes, and project-specific validation evidence. development-methodology, create-agents-plan, documentation-bootstrap, and documentation-page-verifier.
User asks for code change Use Coding Agent unless the task is primarily review, verification, documentation, or merge coordination. careful-coding, code-comments, code-discovery, plus every folder technology skill declared by the nearest AGENTS.md.
User asks for review Use Code Review Agent with fresh context and finding-first output. Extract evidence from the applicable coding-skill checklists before synthesis. code-review-evidence, review-structured, code-comments, and every folder technology skill declared by the nearest AGENTS.md.
User asks for documentation artifact Use Documentation Writer and development-methodology to select exactly one non-wiki creation route and write the source-backed artifact. development-methodology plus create-functional-spec, create-architecture, create-high-level-design, create-module-design, or create-unit-test-plan.
User asks to set up project agents or skills Use Project Agent Setup Agent to create AGENTS-PLAN.yaml and decide whether nested plans are needed. detect-technology-skills, development-methodology, create-agents-plan, documentation-page-verifier, and documentation-bootstrap for first-time setup.
User asks for artifact review Use Artifact Review Agent with the matching non-wiki artifact review skill and completed checklist evidence. review-architecture, review-functional-spec, review-high-level-design, review-module-design, or review-unit-test-plan.
User asks for wiki setup or restructuring Use Wiki Setup Agent to establish roots, ownership, federation, source workflow, navigation, and setup verification. project-wiki, with create-project-wiki or documentation-bootstrap only when their request-specific conditions apply.
User asks for wiki content or ingest Use Wiki Writer Agent for ordinary durable maintenance or Wiki Ingest Agent for raw-to-processed synthesis. project-wiki, project-wiki-topic-writer, project-wiki-topic-verifier, and code-project-wiki only for code-aware maintenance.
User asks a wiki-backed question or requests research Use Wiki Query Agent for maintained knowledge and Wiki Research Agent or Public Source Collector when missing coverage requires a raw-only source artifact. project-wiki-query and project-wiki-research, with collection kept separate from durable synthesis.
User asks for wiki verification or review Use Wiki Topic Verifier for changed topic pages or Wiki Artifact Reviewer for a project-wiki methodology artifact. project-wiki-topic-verifier, or review-project-wiki plus documentation-page-verifier.
Target files are in a specialized subtree Read the nearest nested AGENTS.md and load every technology skill declared for the subtree before acting. Do not rerun detection. Project Agent Setup owns source-backed detection and blocks missing required skills or exclusive conflicts before publishing the guidance.
Work spans multiple phases or agents Use Development Orchestrator to split the work and preserve handoff, claim, and verification records. structured-design, manage-backlog, agent-claim, agent-work-merge, review-structured.

Customer Deployment Contracts

AGENTS.md Is Local Reference

Root and nested AGENTS.md files select project family, tiers, paths, validation commands, and data boundaries. They do not duplicate shared skills.

Skills Stay Reusable

Skills contain portable procedures. They are loaded by role and task, not copied into project instructions or role prompts.

Roles Keep Authority Stable

New roles are added only when authority, output contract, lifecycle phase, context isolation, or runtime resource ownership changes.

Evidence Is Required

Reviews, verification, handoffs, and wiki updates should leave enough evidence for another agent or human to inspect the decision later.

Customer Names Stay Stable

Customer maintainers may edit installed skills and native agent definitions directly. Keep the original names so loadouts and cross-references remain valid.

Updates Are Reconciled

A later update compares the old generic, installed customer, and new generic definitions. The user chooses what to keep, merge, replace, or remove.
