Agent And Skill Specialization Strategy

This page owns the rules for deciding when work needs a role, a fixed or request-specific skill, or a setup-time folder skill. The generated catalog owns the current inventory, the operating model owns project contracts, the orchestrated development lifecycle owns execution order, and runtime formats owns adapter behavior.

Selection Model

Selection Owner When it applies Maintenance boundary
Role agent Canonical role source The work needs a stable responsibility, authority boundary, context boundary, or output contract. Current roles, loadouts, and outputs appear only in the generated catalog.
Fixed role skill Canonical role skill entry without a condition Every invocation of the role needs the expertise. The role always loads it.
Request-specific skill Canonical role skill entry with a human-readable condition The requested outcome and available evidence satisfy the condition. The role applies the condition with judgment; wording alone is not the decision boundary.
Folder technology skill Skill-owned detection metadata and project configuration Setup-time repository evidence selects the skill for a folder scope. The harness supplies the established loadout; ordinary roles do not rerun detection.

Deterministic Setup-Time Technology Detection

Generic canonical loadouts

Canonical roles contain stable responsibility skills only. Test frameworks, interface frameworks, identity providers, persistence libraries, orchestration runtimes, and optional analysis tools are not fixed into generic agents.

Skill-owned activation

Each specialized technology or domain skill owns detection.yaml with an explicit skill identifier and any-of activation branches. A branch may require all of several file, same-file path and extension, owning manifest, dependency, content, or parsed source-import predicates. Generic fixed-role skills and optional local tools are absent from detection metadata.

Generated registry

scripts/build-technology-detection.py validates every specialized variant, rejects task-time fields and specialized fixed role skills, generates the portable registry, and verifies the installed detector mirror against the canonical script.

Detector ownership

When the operating lifecycle requires configuration, Project Configurator selects representative folder scopes, runs detection once, records source-backed loadouts in PROJECT.yaml, and renders unconditional folder skill-loading instructions into AGENTS.md. Ordinary agents consume that established loadout instead of rerunning detection.

Truthful proof boundary

The detector selects a named skill only when one complete activation branch succeeds. It records concrete scope evidence, installed availability, missing required skills, exclusive conflicts, and explicit no-variant results. Runtime activation remains a separate harness-verification concern.

Stable inputs

Agent role, task wording, prompt keywords, skill-read confirmation, and optional local commands are not detector inputs. Setup-selected folder scopes and repository evidence are the correctness boundary.

When To Create A Specialized Agent

Decision Create a specialized agent when Use a routed skill when Example
Authority The role has different write permissions, read-only behavior, approval rights, or escalation authority. The same executor can safely use a different framework or language rule. Dev Code Reviewer is separate. React coding is a Dev Coder skill pack.
Output contract The role returns a different artifact shape, such as a review verdict, QA report, design spec, or path decision. The role still returns the same patch, explanation, or verification shape. Dev Verifier is separate. Jest versus Vitest is a test skill choice.
Context isolation The work benefits from an independent context, adversarial reading, or clean-room validation. The same context can safely continue with a stack-specific procedure. Dev Artifact Reviewer is separate. API route coding is not separate.
Runtime resource The role owns exclusive resources such as browser sessions, E2E servers, shared databases, or merge lanes. The task only changes which test command or runtime library applies. Dev Browser Operator can be separate. Playwright knowledge is also a skill.
Lifecycle phase The role belongs to a different phase: orchestrate, plan, implement, review, verify, merge, or document. The same phase changes language, stack, folder, or artifact subtype. Dev Orchestrator is separate. TypeScript and Java coding are skills.
