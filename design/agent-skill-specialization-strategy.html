Agent And Skill Specialization Strategy

The methodology has a shared catalog of role agents and skills that every project can use. AGENTS.md files provide the project-local reference that lets those shared skills route work without forking the common role source. Role agents act as skill routers: they read caller parameters and AGENTS.md reference, classify the work, load the smallest relevant skill set, and produce the role's contracted output.

Specialization Principle

Agents Specialize By Responsibility

Agents remain broad, invokable roles with stable behavior and clear authority boundaries.

  • Development Orchestrator decomposes and routes work.
  • Coding Agent edits code across stacks through skill dispatch.
  • Code Review Agent reviews with fresh context and read-only authority.
  • QA And Verification Agent proves outcomes with commands and evidence.
  • Project Agent Setup Agent creates AGENTS-PLAN.yaml for role and skill routing setup.
  • Project Organiser owns file placement and project taxonomy.

Skills Specialize By Expertise

Skills stay small and composable, so a single role can load the right knowledge for a specific job without carrying unrelated guidance.

  • Stack skills cover React, Next.js, Electron, Spring Boot, Python, Drizzle, Postgres, or MLX.
  • Artifact skills cover API routes, UI components, prompts, CLI commands, tests, docs, and wiki pages.
  • Quality skills cover testing, security, accessibility, privacy, performance, and observability.
  • Project skills cover local rules, folder taxonomy, commands, generated files, and data boundaries.

Configuration Layers

development-methodology Skill

The development-methodology skill is the shared router for documentation work. It does not know the target repository by itself; it defines how to classify a documentation request once repository facts are available.

  • It names the companion skills for setup, reverse engineering, project wiki work, artifact creation, artifact review, and page verification.
  • It enforces loading discipline: load the routing skill, then exactly one matching creation skill for a normal artifact creation job.
  • It chooses the document route: AGENTS-PLAN.yaml, project wiki, functional specification, architecture, high-level design, or module design.
  • It provides the shared page contract and verification expectations so every routed artifact has the same durable structure.

AGENTS-PLAN.yaml

AGENTS-PLAN.yaml is the setup and validation artifact for project agent roles, skill loadouts, folder routing, nested plan placement, and customer-safe example boundaries.

  • It is created with create-agents-plan before AGENTS.md routing guidance is written or refreshed.
  • It records source evidence for role and skill choices, including proprietary validation notes when the file lives inside a private project.
  • It can exist at the project root and inside subfolders with distinct technology, runtime, data, or verification needs.
  • It is a planning artifact, not the normal runtime reference loaded for every task.

Root AGENTS.md

The root AGENTS.md supplies the project-specific reference that the skills router needs before it can apply shared routing rules to this repository.

  • It declares the project family, application tiers, important paths, validation commands, privacy rules, and local terminology.
  • It tells role agents which source roots, test roots, wiki roots, procedure files, and taxonomy pages matter first.
  • It can say which skills normally apply to Front-end, API, static pages, DB, documentation, or other local tiers.
  • It selects local reference and policy; it does not copy the full body of shared skills or redefine role authority.

Nested AGENTS.md

Nested AGENTS.md files narrow the root reference for a subtree. They are useful when one folder has a different tier, stack, data rule, generated-file policy, or verification command.

  • <subtree>/AGENTS.md applies to work inside that subtree.
  • The nearest AGENTS.md adds local reference after the root AGENTS.md has established the repository-wide model.
  • Nested AGENTS.md files should contain only local differences, such as API contract checks or DB migration rules.
  • They route skills and local checks; they do not create folder-specific agents.

Fictitious Example

Acme Ledger is a web application with Front-end, API, static pages, and DB tiers. Its root AGENTS.md says API work lives under src/app/api and DB work lives under src/db.

  • The user asks for a design for invoice retry behavior.
  • The Documentation Architect loads development-methodology because the request is documentation work.
  • development-methodology classifies the artifact as a high-level design because retry behavior crosses API routes, DB state, and user-visible status.
  • The skills router reads AGENTS.md and sees that the relevant folders imply api-routes, postgres-drizzle, typescript-strict, and jest guidance.
  • If src/db/AGENTS.md says migration changes need a rollback note, that local rule is included in the design and verification checklist.
  • This works because shared routing answers what kind of artifact to create, while AGENTS.md answers which project facts and tier-specific skills make that route correct here.

Shared Catalog Lifecycle

Shared installs are synchronized from this source catalog and keep an ownership boundary for cleanup.

  • scripts/refresh-shared-skills.py replaces the shared machine installs from the current source tree.
  • .dev-methodology-install.json records the skills this catalog installed into each shared destination.
  • Before a rename or deletion, sweep the source catalog for the old skill id and update role definitions, Codex metadata, skills, dispatch profiles, workflow examples, design documents, scripts, and tests.
  • Creation skills keep one artifact template route each and load only when that artifact type is the job at hand.
  • Review skills keep checklist templates named review-checklist-review-target.md and completed checklist artifacts named artifact-name.review-checklist-review-target.md.
  • Renamed or deleted skills are pruned only when the ownership manifest says this catalog installed the old artifact.
  • Runtime agent metadata packaged inside a skill is refreshed and pruned with that owned skill.
  • The first refresh without a manifest writes the baseline and skips pruning so unowned local skills stay untouched.

Role Agent Dispatch Loop

1. Read request

Parse caller parameters: role, task intent, target files, constraints, requested verification, and allowed authority.

2. Load AGENTS.md reference

Read the root AGENTS.md, nearest nested AGENTS.md, local instructions, package metadata, wiki map, and relevant procedures.

3. Classify work

Determine project family, application tier, artifact type, language, runtime container, data sensitivity, and validation surface.

4. Select skills

Load common role skills, project skills, stack skills, artifact skills, quality skills, and folder-specific skills.

5. Execute contract

Do the role's job with the selected skills only, then produce the role output and verification evidence.

When To Create A Specialized Agent

Decision Create a specialized agent when Use a routed skill when Example
Authority The role has different write permissions, read-only behavior, approval rights, or escalation authority. The same executor can safely use a different framework or language rule. Code Review Agent is separate. React coding is a Coding Agent skill pack.
Output contract The role returns a different artifact shape, such as a review verdict, QA report, design spec, or path decision. The role still returns the same patch, explanation, or verification shape. QA And Verification Agent is separate. Jest versus Vitest is a test skill choice.
Context isolation The work benefits from an independent context, adversarial reading, or clean-room validation. The same context can safely continue with a stack-specific procedure. Artifact Review Agent is separate. API route coding is not separate.
Runtime resource The role owns exclusive resources such as browser sessions, E2E servers, shared databases, or merge lanes. The task only changes which test command or runtime library applies. E2E Browser Agent can be separate. Playwright knowledge is also a skill.
Lifecycle phase The role belongs to a different phase: orchestrate, plan, implement, review, verify, merge, or document. The same phase changes language, stack, folder, or artifact subtype. Development Orchestrator is separate. TypeScript and Java coding are skills.

Root AGENTS.md Shape

Repository contract

The root AGENTS.md is the repository's contract with the shared catalog. Agent routers load it after host safety rules and before nested AGENTS.md files.

  • Project family and application tiers tell the router what kind of system it is handling.
  • AGENTS-PLAN.yaml is the setup artifact that explains why these routing choices exist.
  • Tier-to-path mappings tell the router which local paths imply Front-end, API, static pages, DB, or other guidance.
  • Skills router notes name the shared skills normally loaded for those tiers.
  • Validation and data rules tell role agents which checks and boundaries apply after work is done.
  • Local taxonomy links give the Project Organiser and Documentation Architect the vocabulary for placement and artifacts.

Local reference forms

Local reference is explicit and scoped. It adapts shared skill routing without mutating the shared skill files.

  • Select: name the skills normally used for a tier or artifact family.
  • Narrow: restrict routing to a path, artifact type, stack, or task intent.
  • Add evidence: point agents to wiki pages, procedures, source roots, and test roots.
  • Add checks: name build, unit, E2E, contract, or migration commands.
  • Set boundaries: declare PII, company-internal data, generated files, or approval rules.

Customer Installation And Direct Customization

Build and install boundary

The source build contains customer-independent skills, canonical roles, documentation data, support scripts, and generated runtime adapters. On the customer machine, the setup agent selects the active runtime and invokes deterministic installers.

Stable-name customization

Customer maintainers may edit an installed skill or native agent definition directly. Preserve its original name so skills, loadouts, and AGENTS.md guidance keep resolving it.

Deliberate upgrades

A refresh must not silently replace customized files. An update uses three-way discrepancy analysis across the old generic, installed customer, and new generic definitions before the user chooses what to keep, merge, replace, or remove.

Project grounding

AGENTS-PLAN.yaml explains selected roles and skills. Root and nested AGENTS.md files contain operational project guidance, with thin colocated CLAUDE.md bridges where Claude Code is used.

Nested AGENTS.md Shape

Folder scope

A nested AGENTS.md applies to its subtree and is read after the root AGENTS.md.

  • It can add path-local stack skills.
  • It can select path-local validation commands.
  • It can mark data sensitivity or generated-file status.
  • It can route documentation, prompt, API, frontend, or test artifacts differently.

Folder constraints

A nested AGENTS.md narrows routing. It does not fork the role catalog.

  • It cannot change an agent's write authority.
  • It cannot change a review verdict schema.
  • It cannot bypass the completed checklist evidence record for review roles.
  • It cannot disable host safety or project data rules.
  • If authority or output contract changes, define a specialized role in the shared catalog or project role directory.

Agent Set Normalization

Project internals are domains

  • If the project implements agents, tools, prompts, model runtimes, schedulers, or harnesses, classify those as application domains first.
  • Coding Agent loads skills for those domains when editing product code, but the domains do not become invocable development agents by default.
  • Create a specialized development agent only when the human or orchestrator needs a separate context, authority model, or output contract.

Generic agents retained

  • Development Orchestrator.
  • Project Organiser.
  • Coding Agent.
  • Code Review Agent.
  • QA And Verification Agent.
  • Documentation Architect.
  • Artifact Review Agent.
  • Wiki Query Agent and Wiki Ingest Agent.
  • Backlog Steward and Merge Coordinator.

Specialized agents allowed

  • E2E Browser Agent for Playwright, browser state, dev server claims, and authenticated workflows.
  • UX Designer or UX Reviewer for design artifacts and independent usability review.
  • Security Reviewer for threat, data, supply-chain, and exploitability review.
  • Runtime Diagnostician for long-running local runtime failures with logs and exclusive resources.
  • Prompt Contract Reviewer for prompt protocol, tool-call safety, and evaluation review.

Specialized agents avoided

  • Frontend Coder, API Coder, Java Coder, Python Coder, React Coder, and Next.js Coder.
  • Framework-specific reviewers when the review output contract is unchanged.
  • Test-framework-specific agents when the job is ordinary code verification.
  • Folder-specific agents whose only difference is path or stack selection.
