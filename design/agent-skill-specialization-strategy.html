Agent And Skill Specialization Strategy

The methodology has a shared catalog of role agents and skills that every project can use. A project-specific definition selects, overrides, dispatches, and aggregates that shared catalog without forking the common role source. Role agents act as skill routers: they read caller parameters and project configuration, classify the work, load the smallest relevant skill set, and produce the role's contracted output.

Specialization Principle

Agents Specialize By Responsibility

Agents remain broad, invokable roles with stable behavior and clear authority boundaries.

  • Development Orchestrator decomposes and routes work.
  • Coding Agent edits code across stacks through skill dispatch.
  • Code Review Agent reviews with fresh context and read-only authority.
  • QA And Verification Agent proves outcomes with commands and evidence.
  • Project Organiser owns file placement and project taxonomy.

Skills Specialize By Expertise

Skills stay small and composable, so a single role can load the right knowledge for a specific job without carrying unrelated guidance.

  • Stack skills cover React, Next.js, Electron, Spring Boot, Python, Drizzle, Postgres, or MLX.
  • Artifact skills cover API routes, UI components, prompts, CLI commands, tests, docs, and wiki pages.
  • Quality skills cover testing, security, accessibility, privacy, performance, and observability.
  • Project skills cover local rules, folder taxonomy, commands, generated files, and data boundaries.

Configuration Layers

Shared Catalog

The shared catalog is the default source for reusable agents, reusable skills, base dispatch rules, and host adapter templates.

  • agents/roles/*.role.yaml defines portable role agents.
  • skills/*/SKILL.md defines portable skills.
  • profiles/base/*.yaml defines common project families, tiers, and stack packs.
  • adapters/* maps shared roles and skills into runtime-specific files.

Project Definition

The project definition selects and adapts the shared catalog for a repository.

  • .agents/project-profile.yaml selects project family, tier, stacks, roles, skill packs, commands, and data rules.
  • docs/project-taxonomy.md defines project-specific file placement and artifact categories.
  • Project-specific definitions can append skills, narrow dispatch rules, override validation commands, and aggregate workflows.
  • Project-specific definitions cannot silently change a shared role's authority or output contract.

Folder Definition

Folder definitions add local routing for subtrees that use different stacks, generated artifacts, data rules, or verification commands.

  • <subtree>/.agents/folder-profile.yaml governs the subtree where it lives.
  • The nearest folder profile applies after the root project profile.
  • Folder definitions contain only local differences from the root profile.
  • Folder definitions are for routing and narrowing, not for creating folder-specific agents.

Shared Catalog Lifecycle

Shared installs are synchronized from this source catalog and keep an ownership boundary for cleanup.

  • scripts/refresh-shared-skills.py replaces the shared machine installs from the current source tree.
  • .dev-methodology-install.json records the skills this catalog installed into each shared destination.
  • Before a rename or deletion, sweep the source catalog for the old skill id and update role definitions, adapter metadata, skills, dispatch profiles, workflow examples, design documents, scripts, and tests.
  • Review skills keep checklist templates named review-checklist-review-target.md and completed checklist artifacts named artifact-name.review-checklist-review-target.md.
  • Renamed or deleted skills are pruned only when the ownership manifest says this catalog installed the old artifact.
  • Runtime agent metadata packaged inside a skill is refreshed and pruned with that owned skill.
  • The first refresh without a manifest writes the baseline and skips pruning so unowned local skills stay untouched.

Role Agent Dispatch Loop

1. Read request

Parse caller parameters: role, task intent, target files, constraints, requested verification, and allowed authority.

2. Load config

Read the root project profile, nearest folder profile, local instructions, package metadata, wiki map, and relevant procedures.

3. Classify work

Determine project family, tier, artifact type, language, runtime container, data sensitivity, and validation surface.

4. Select skills

Load common role skills, project skills, stack skills, artifact skills, quality skills, and folder-specific skills.

5. Execute contract

Do the role's job with the selected skills only, then produce the role output and verification evidence.

When To Create A Specialized Agent

Decision Create a specialized agent when Use a routed skill when Example
Authority The role has different write permissions, read-only behavior, approval rights, or escalation authority. The same executor can safely use a different framework or language rule. Code Review Agent is separate. React coding is a Coding Agent skill pack.
Output contract The role returns a different artifact shape, such as a review verdict, QA report, design spec, or path decision. The role still returns the same patch, explanation, or verification shape. QA And Verification Agent is separate. Jest versus Vitest is a test skill choice.
Context isolation The work benefits from an independent context, adversarial reading, or clean-room validation. The same context can safely continue with a stack-specific procedure. Artifact Review Agent is separate. API route coding is not separate.
Runtime resource The role owns exclusive resources such as browser sessions, E2E servers, shared databases, or merge lanes. The task only changes which test command or runtime library applies. E2E Browser Agent can be separate. Playwright knowledge is also a skill.
Lifecycle phase The role belongs to a different phase: orchestrate, plan, implement, review, verify, merge, or document. The same phase changes language, stack, folder, or artifact subtype. Development Orchestrator is separate. TypeScript and Java coding are skills.

Project Definition Shape

Root profile

The root profile is the repository's contract with the shared catalog. Agent routers load it after host safety rules and before folder profiles.

  • extends imports shared catalog versions.
  • agents enables shared roles and adds project-specific startup reads or common skills.
  • skills selects shared packs and defines project-local packs.
  • application_domains names product internals that development agents may need to understand.
  • dispatch maps paths, artifact types, and task intents to skills.
  • aggregates names multi-agent workflows without creating new agent types.

Override forms

Overrides are explicit and scoped. They adapt the shared catalog without mutating it.

  • Append: add project skills, reads, checks, or commands to a shared role.
  • Narrow: restrict a rule to a path, artifact type, stack, or task intent.
  • Replace: substitute a project-owned skill pack for a shared pack by name.
  • Disable: turn off a shared pack or role with an explicit reason.
  • Aggregate: define a named workflow that invokes several agents and skill packs in order.
version: 1

extends:
  catalog: dev-methodology
  roles_version: current
  skills_version: current

project:
  family: full-stack-product
  tier: greenfield-product
  stacks:
    - nextjs-app-router
    - typescript
    - postgres
  application_domains:
    - questionnaire-workflow
    - analysis-engine
    - authenticated-user-experience

agents:
  enabled:
    - development-orchestrator
    - project-organiser
    - coding-agent
    - code-review-agent
    - qa-and-verification-agent
    - documentation-architect
  overrides:
    coding-agent:
      append_startup_reads:
        - AGENTS.md
        - package.json
      append_common_skills:
        - project-local-coding-rules
    qa-and-verification-agent:
      validation_commands:
        build: pnpm run build
        unit: pnpm test

skills:
  packs:
    project-local-coding-rules:
      includes:
        - careful-coding
        - ast-grep
        - review-structured
    nextjs-api-work:
      includes:
        - typescript-strict
        - nextjs-app-router
        - api-routes
        - clerk-auth
        - jest

dispatch:
  coding-agent:
    common_skills:
      - careful-coding
      - ast-grep
    rules:
      - match: src/app/api/**/route.ts
        artifact: api-route
        add_skills:
          - nextjs-api-work
      - match: src/components/**
        artifact: frontend-ui
        add_skills:
          - react-server-components
          - tailwind-design-system
          - playwright

aggregates:
  verified-code-change:
    sequence:
      - agent: project-organiser
        when: creates_or_moves_files
      - agent: coding-agent
      - agent: code-review-agent
      - agent: qa-and-verification-agent
    final_output: verified_patch_summary

Folder Definition Shape

Folder scope

A folder profile applies to its subtree and is read after the root project profile.

  • It can add path-local stack skills.
  • It can select path-local validation commands.
  • It can mark data sensitivity or generated-file status.
  • It can route documentation, prompt, API, frontend, or test artifacts differently.

Folder constraints

A folder profile narrows routing. It does not fork the role catalog.

  • It cannot change an agent's write authority.
  • It cannot change a review verdict schema.
  • It cannot bypass the completed checklist evidence record for review roles.
  • It cannot disable host safety or project data rules.
  • If authority or output contract changes, define a specialized role in the shared catalog or project role directory.
scope:
  path: src/app/api
  artifact_family: api

dispatch:
  coding-agent:
    append_common_skills:
      - api-routes
      - clerk-auth
      - typescript-strict
      - jest

validation:
  focused:
    - pnpm test -- src/app/api
  always_after_code:
    - pnpm run build

data_rules:
  pii_to_external_llm: forbidden
  log_sensitive_payloads: forbidden

Agent Set Normalization

Project internals are domains

  • If the project implements agents, tools, prompts, model runtimes, schedulers, or harnesses, classify those as application domains first.
  • Coding Agent loads skills for those domains when editing product code, but the domains do not become invocable development agents by default.
  • Create a specialized development agent only when the human or orchestrator needs a separate context, authority model, or output contract.

Generic agents retained

  • Development Orchestrator.
  • Project Organiser.
  • Coding Agent.
  • Code Review Agent.
  • QA And Verification Agent.
  • Documentation Architect.
  • Artifact Review Agent.
  • Wiki Query Agent and Wiki Ingest Agent.
  • Backlog Steward and Merge Coordinator.

Specialized agents allowed

  • E2E Browser Agent for Playwright, browser state, dev server claims, and authenticated workflows.
  • UX Designer or UX Reviewer for design artifacts and independent usability review.
  • Security Reviewer for threat, data, supply-chain, and exploitability review.
  • Runtime Diagnostician for long-running local runtime failures with logs and exclusive resources.
  • Prompt Contract Reviewer for prompt protocol, tool-call safety, and evaluation review.

Specialized agents avoided

  • Frontend Coder, API Coder, Java Coder, Python Coder, React Coder, and Next.js Coder.
  • Framework-specific reviewers when the review output contract is unchanged.
  • Test-framework-specific agents when the job is ordinary code verification.
  • Folder-specific agents whose only difference is path or stack selection.
