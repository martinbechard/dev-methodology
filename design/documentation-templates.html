Documentation Templates

The development methodology ships seven source-backed templates for project configuration, wiki methodology, product behavior, system design, module design, and unit test planning. Each template supplies TODO instructions that are replaced with project evidence when an artifact is created.

Template Catalog

Project Configuration

Records the complete project-wide intent for conceptual agent definitions, technology skillsets, folder routes, AGENTS.md locations, validation evidence, and customer-safe examples.

  • Output: PROJECT.yaml at the project root
  • Creator: create-project-configuration
  • Verifier: documentation-page-verify
project-template.yaml

Project Wiki Methodology

Defines a project-specific wiki root, authority order, base page contract, topic organization, diagrams, source links, maintenance workflow, automation boundaries, and verification.

  • Output: project wiki methodology artifact
  • Creator: project-wiki-create
  • Reviewer: project-wiki-review
project-wiki-template.md

Functional Specification

Describes actor goals, entry points, user-visible workflows, states, permissions, edge cases, acceptance behavior, and workflow-level verification.

  • Scope: one coherent user or operator capability
  • Creator: create-functional-spec
  • Reviewer: review-functional-spec
functional-spec-template.md

Architecture

Defines project-wide boundaries, system context, technology stack, file ownership, dependency layers, major components, cross-cutting concerns, invariants, risks, and verification.

  • Scope: a system or cross-cutting architectural concern
  • Creator: create-architecture
  • Reviewer: review-architecture
architecture-template.md

High-Level Design

Explains a subsystem, feature family, or system slice through requirements coverage, constituent components, interactions, data contracts, lifecycle, configuration ownership, implementation order, and readiness.

  • Scope: a coherent multi-module design
  • Creator: create-high-level-design
  • Reviewer: review-high-level-design
high-level-design-template.md

Module Design

Specifies one implementation unit, including its responsibility, callers, dependencies, contracts, state, processing rules, interfaces, errors, tests, and implementation readiness.

  • Scope: one module, service, class, task, utility, or UI component
  • Creator: create-module-design
  • Reviewer: review-module-design
module-design-template.md

Unit Test Plan

Maps one unit's observable behavior, collaborators, boundary doubles, failures, invariants, and coverage gaps to stable source-backed test scenarios.

  • Scope: one testable implementation boundary
  • Creator: create-unit-test-plan
  • Reviewer: review-unit-test-plan
unit-test-plan-template.md

Template Selection

Choose the smallest artifact that fully owns the requested knowledge. Project configuration and wiki methodology establish operating context; the remaining templates form a progressively narrower development chain.

Question to answer Template Relationship to other artifacts
Which agents, skills, folder rules, and guidance files apply to this repository? Project Configuration Establishes setup intent for later agent work; it is not a product or implementation design.
How should durable project knowledge be organized, sourced, maintained, and verified? Project Wiki Methodology Defines the synthesis layer that links the authoritative specifications, designs, source, tests, and backlog.
What must a user, operator, administrator, service, or external system be able to accomplish? Functional Specification Supplies accepted behavior and workflows to architecture and downstream design.
What system-wide structure, boundaries, stack, and dependency rules govern implementation? Architecture Constrains high-level designs while leaving individual subsystem detail to them.
How do several components collaborate to deliver a subsystem or feature family? High-Level Design Accounts for accepted requirements and divides them into implementable module responsibilities.
What exact contract, state, processing, dependencies, and error behavior belong to one unit? Module Design Provides the implementation-ready boundary consumed by coding and test work.
Which scenarios prove one unit's behavior, boundaries, failures, and invariants? Unit Test Plan Translates accepted contracts and implementation evidence into durable test obligations.

Wiki Format

A project wiki lives under docs/wiki and acts as compiled understanding rather than primary authority. The wiki methodology template records project-specific maintenance rules; the project-wiki page schema owns the concrete format for topic pages, folders, and required root files.

Wiki repository shape

  • README.md introduces the wiki and its primary navigation.
  • schema.md records authority, page sections, and maintenance rules.
  • topic-index.md maps topics and their source neighborhoods.
  • glossary.md defines recurring project terminology.
  • open-decisions.md and known-defects.md retain unresolved and defective states.
  • maintenance-log.md records notable maintenance events.
  • Topic folders use index.md hub pages plus granular leaf pages.
  • digests/index.md and YYYY-MM.md pages summarize ingested developments when digests apply.

Shared page contract

  1. Current Understanding
  2. Authoritative Sources
  3. Related Code
  4. Related Tests
  5. Related Backlog Items
  6. Related Wiki Pages
  7. Open Questions
  8. Maintenance Notes

Page organization

  • Non-reserved concept pages use YAML frontmatter with a non-empty type; title, description, and tags improve routing and previews.
  • Folder index.md and log.md files are reserved navigation pages and do not need concept frontmatter.
  • Hub pages describe a stable domain and link to granular leaves that can change or be maintained independently.
  • Additional topic-specific sections may follow the shared contract without replacing its required sections.
  • Specialized methodology documents keep the eight shared sections first and append the extension sections owned by their template.

Format boundary: the shared wiki-compatible contract applies to docs/wiki topic pages and methodology artifacts created from these templates. It does not automatically apply to README files, design HTML pages, runtime schemas, generated data, or native agent definitions that already have another authoritative format.

Read the complete project wiki page schema for leaf splitting, digest ordering, source-link, OKF, and maintenance rules.

Template Completion

Source-backed content

  • Copy only the template that matches the requested artifact.
  • Replace every TODO instruction with project-specific evidence.
  • Link authoritative sources, code, tests, backlog records, and related pages where the explanation depends on them.
  • Use Not yet identified when a required source, implementation, test, or related record does not exist.

Review and maintenance

  • Run the matching artifact reviewer after the document is complete.
  • Keep steady-state prose and record unresolved decisions explicitly.
  • Add editable Mermaid diagrams only when a real relationship is easier to understand visually.
  • Recheck the document when its sources, responsibilities, contracts, or verification evidence change.
