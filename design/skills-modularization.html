Technology Skills

The toolkit separates stable, technology-agnostic ways of working from the extensions needed for a specific stack. An agent definition owns the skills it always needs and the conditional skills it uses only when a stated rule matches. During project setup, Project Configurator binds compatible technology skills to folders from repository evidence.

Core Agent Skills

An agent definition may directly reference both required and optional core skills. A required skill has no condition and loads whenever that agent runs. An optional skill has a human-readable condition and loads only when the request and available evidence make it applicable. Both kinds of directly referenced skills must be technology-agnostic so the generic agent definition can be reused across programming languages, frameworks, project types, and agentic coding vendors. View their complete definitions in the Skill Catalog on the Core Agent and Skills page.

Skill relationship How it is declared When it loads Example
Required core skill A skill entry in the agent definition without a condition. Every invocation of that agent needs the capability. Dev Coder always uses careful-coding, code-comments, code-discovery, and fix-explanation.
Optional core skill A skill entry in the agent definition with a human-readable condition. The request and available evidence satisfy the rule; the agent applies judgment rather than matching prompt words. Dev Coder uses test-driven-development when behavior can be expressed through executable tests.
Loading transport is separate from skill selection

Codex or Junie remains responsible for inserting selected skill instructions into agent context. A skill already preloaded by the harness is not reread through MCP. When routing has selected several exact names but their content is not yet present, the optional mcp-agent-ops layer loads the complete set in one bounded skill_load call and loads only required supporting resources afterward. Catalog revisions and content digests identify supplied bytes; they do not prove that a model retained or followed them.

The project-configuration evaluation keeps runtime availability separate from instruction treatment. It stages effective instructions and exact approved MCP resources only for execution skills and frontmatter-only entries for other available skills, then uses a session-bound evaluator audit to select the one parent or custom-agent process stream that satisfies required MCP sequences. The template resource used by the case is absent from preloaded harness context, so the required load proves the resource transport. Every required operation binds a privacy-safe argument digest and bounded outcome to the same completed call. Exact configuration, catalog, authorization, and allowed output evidence survives disposable-workspace cleanup. Codex uses a digest-bound permission profile for the Git lifecycle, with network, host-home, and evidence-root access denied and writes limited to the disposable workspace, its Git metadata, and its temporary directory; other system paths remain read-only. Junie uses an owner-only action policy that permits the same exact MCP operations plus Git add and Git commit; this controls capabilities but does not establish filesystem containment. Skill-probe variants disable the catalog so an omitted treatment skill cannot reappear through a tool.

Why Technology-Specific Skills Are Loaded Separately

Required versus optional is not the architectural split: both are valid forms of technology-agnostic skill references inside an agent definition. The actual boundary is technology dependence. No finite agent definition can list every language, framework, library, runtime, database, and tool that may exist now or be added later. Listing technology skills in generic agents would also require every relevant agent definition to be changed whenever the technology catalog grows.

The toolkit therefore uses two independent loading mechanisms. Agent definitions provide their own required and optional technology-agnostic skills. Project setup separately considers only technology and domain skills that actually exist in the available skill collection, detects which ones match each project or folder scope, and binds those skills through PROJECT.yaml and AGENTS.md. New technology skills can be added without editing every generic agent definition.

Loading mechanism What it can load How selection works Why it belongs here
Agent-definition loading Required and optional technology-agnostic skills declared directly in the conceptual agent definition. A required entry loads every time. An optional entry loads when its human-readable condition applies to the request and available evidence. These skills describe stable responsibilities and methods that remain useful regardless of the project's technology stack.
Folder technology loading Available technology and domain skills with detection metadata, such as python, fastapi, the separate Java, Spring Boot, and Quarkus coding and design skills, or the focused JHipster skills. Project Configurator detects matching repository evidence for each scope, records the intended loadout in PROJECT.yaml, and renders it as unconditional folder guidance in AGENTS.md. The project and folder determine the applicable technologies, and the available technology catalog can grow independently of the generic agents.

Technology Extensions Setup Process

Project Configurator owns the setup process. It inspects the target repository, selects technology extensions from source evidence, records the intended configuration in PROJECT.yaml, renders the operational folder guidance, and verifies that the result matches the recorded intent.

Development precursor — outside setup

Actor: Methodology maintainer. Technology skills and their detection metadata are authored, validated, and shared before setup begins. Project Configurator consumes that existing collection; it does not create technology skills while configuring a target project.

Actor: Project Configurator
1. Inspect project scopes

Inventory the project family, application tiers, manifests, source roots, tests, and folder boundaries without combining unrelated sibling technologies.

Actor: Project Configurator
2. Match technology evidence

Run setup-time detection for representative folders. Accept a technology skill only when one complete evidence rule matches, and retain missing requirements, conflicts, and explicit no-variant results.

Actor: Project Configurator
3. Write the intermediate configuration

Create or update the root PROJECT.yaml before generating operational guidance. This reviewable intent log records the selected agent definitions, folder skillsets, source evidence, routing decisions, and planned AGENTS.md locations.

Actors: Maintainer and Project Configurator
4. Review or correct the intent

The maintainer can edit PROJECT.yaml to force a correction to the intended configuration. Project Configurator then reconciles that edit with repository evidence and bundle constraints before continuing.

Actor: Project Configurator
5. Render folder guidance

Generate root or nested AGENTS.md files with unconditional folder skillsets. For Claude Code, create a thin colocated CLAUDE.md bridge beside each applicable AGENTS.md.

Actor: Project Configurator
6. Verify the setup

Confirm that PROJECT.yaml, folder guidance, bridges, referenced agent definitions, skills, paths, and validation commands agree and that every required runtime capability is available.

Operational result — after setup

Actors: Coding, review, diagnosis, and verification agents. Ordinary work loads the established skillset for the active folder. These agents use the configuration output and do not rerun setup-time detection.

How Setup-Time Technology Detection Works

Project Configurator runs the detector once for each representative folder scope selected during setup. The detector is a repository scanner, not an LLM judgment call: it receives the project root and folder scopes, evaluates versioned rules against files and manifests, and returns a structured result for each scope. It never uses the agent name, prompt wording, task description, or whether an agent says that it read a skill.

What Project Configurator runs

Project Configurator prefers the detect_technology_skills MCP operation when mcp-agent-ops is configured. The server binds the request to its complete active skill catalog and trusted detection registry. The distributed command remains the portability fallback only when the tool is absent or the server cannot initialize or connect before request dispatch. A valid BLOCKED, NO_VARIANT, path, root, authorization, input-policy, or other structured result is never bypassed through the fallback.

Both surfaces receive one project root and one or more scope arguments. A scope can be a folder such as web/src or api/app, or a single file. The detector walks files inside that scope while ignoring dependency, generated-output, and environment directories such as node_modules, dist, target, .next, .venv, and .git. The MCP surface additionally requires the absolute project root and every discovered source or owner file to resolve beneath an administrator-configured workspace root.

python3 [detector-skill]/scripts/detect.py --project-root [root] --scope [folder]

For each scoped file, the detector walks upward to the nearest supported owning project boundary. Supported owner files include package.json, pyproject.toml, requirements files, pom.xml, and Gradle build files. This prevents a child package from inheriting an unrelated dependency from a parent workspace. If one requested scope crosses multiple owning project directories, detection stops with a BLOCKED result and asks Project Configurator to analyze those owners as separate scopes.

Where the detection rules come from

Every bundled technology or domain skill has a detection.yaml file beside its SKILL.md. A build step validates those files and combines them into the generated detection registry consumed by the detector. Each rule names the skill it can select, the evidence that is sufficient, any companion skills that must be added, and whether the skill competes with alternatives in an exclusive group.

For example, the FastAPI rule succeeds when the scope contains Python code and the nearest owning manifest declares fastapi, or when parsed Python code contains a real FastAPI import. If FastAPI is selected, its python companion skill is added to the same loadout. A comment or string containing an example FastAPI import does not count.

How one rule decides whether it matches

Every rule has a root anyOf list. At least one branch in that list must succeed. A branch can contain an allOf list, in which case every check in that branch must succeed. The detector keeps the concrete file, manifest, dependency, or import that satisfied the branch so Project Configurator can review why a skill was selected.

Rule mechanism What the detector actually checks Why this mechanism exists
fileExtension and fileGlob Look for a real file in the selected scope with a required extension or project-relative path pattern. Establishes that relevant source or configuration files exist instead of inferring a technology from prose.
fileMatch Requires one file to satisfy both the path pattern and the allowed extension. Prevents a matching documentation filename from borrowing the extension of an unrelated source file elsewhere in the scope.
owningDependency and manifestFile Check the nearest owning project boundary for a declared dependency or a required manifest or configuration filename. Dependency names are parsed from package.json, pyproject.toml, and requirements files. Keeps evidence attached to the package that owns the scoped code and prevents sibling or parent projects from contaminating the result.
sourceImport Parses Python syntax in .py or .pyi files and looks for a real import of the named module. Comments and string literals are ignored. Allows source code to prove framework use even when dependency metadata is incomplete, without accepting example text as executable evidence.
contentPattern Looks for an exact text marker inside a file whose path matches the rule's glob. Supports configuration formats where a meaningful feature is expressed by file content rather than a dependency name.

How selected skills, companions, and conflicts are resolved

Most detected skills are additive, so every matching skill is kept. A selected skill can also require companion skills; FastAPI, for example, adds python even if the Python rule was not independently selected. For an exclusive group, the lowest numeric priority wins. If two matching alternatives have the same winning priority, the detector does not guess: it returns an exclusive conflict and marks the scope BLOCKED. It also marks the scope BLOCKED when a required selected or companion skill is not available in the installed skill collection.

What the detector returns

Status Meaning What Project Configurator does next
READY The scope has a usable skill loadout. The result lists the path pattern, selected skills, and source evidence for each selection. Review the evidence and record the accepted loadout in PROJECT.yaml.
NO_VARIANT No bundled technology or domain rule matched the scope. This is an explicit empty result, not a guess or a failure. Record that no bundled variant applies instead of inventing technology support.
BLOCKED The scope is missing, resolves outside the project, spans multiple owners, needs an unavailable required skill, or has an unresolved exclusive conflict. Correct the scope, install or provide the missing skill, or resolve the conflict before generating folder guidance.

After reviewing the result, Project Configurator writes the intended skillsets and their evidence to PROJECT.yaml. It then renders unconditional folder skillsets into root or nested AGENTS.md files. Ordinary coding, review, diagnosis, and verification agents consume that established guidance and never run the detector during normal work. Project Configurator reruns detection only when project setup or a technology boundary changes.
