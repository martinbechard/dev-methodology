Skills Modularization

The kit separates stable, technology-agnostic ways of working from the extensions needed for a specific stack. An agent definition owns the skills it always needs and the conditional skills it uses only when a stated rule matches. During project setup, Project Configurator binds compatible technology skills to folders from repository evidence. Use Agent And Skill Definitions for the generated catalog, Agentic Configuration for vendor runtime files, Generic Agent Definitions Source for source-to-vendor conversion, Examples for project configurations, and the Orchestrated Development Lifecycle for execution gates.

Core Agent Skills

Each agent definition directly references a small set of core skills. A core skill without a condition is always loaded for that agent. A core skill with a human-readable condition is loaded on demand when the request and available evidence satisfy the rule. These direct references are limited to technology-agnostic skills so the same generic agent can work across programming languages, frameworks, project types, and agentic coding vendors. Technology-dependent skills are therefore not referenced directly by generic agents; a secondary setup-time mechanism detects repository evidence and binds the matching shared skills to folders through PROJECT.yaml and AGENTS.md.

Skill relationship How it is declared When it loads Example
Always-loaded core skill A skill entry in the agent definition without a condition. Every invocation of that agent needs the capability. Dev Coder always uses careful-coding, code-comments, code-discovery, and fix-explanation.
On-demand core skill A skill entry in the agent definition with a human-readable condition. The request and available evidence satisfy the rule; the agent applies judgment rather than matching prompt words. Dev Coder uses test-driven-development when behavior can be expressed through executable tests.
Technology-specific extension Detection metadata beside the shared skill plus a folder binding in project configuration. Project setup proves that the technology applies to a folder, then ordinary work receives it as an unconditional folder skill. A Python FastAPI folder receives python and fastapi without adding either skill to the generic Dev Coder definition.

Why Core Agent Skills Split Into Two Loading Modes

The split is based on whether a capability is part of the agent's minimum operating contract or is a situational method. Always-loaded skills protect the responsibilities the agent must fulfill on every invocation. On-demand skills are valid extensions of that role, but load only when an explicit condition makes them relevant. This is a loading decision within each agent definition, not a way of categorizing agents.

Always loaded: protect the role

These skills provide safeguards and responsibilities that cannot be omitted without weakening the agent's basic contract. Making them conditional would allow the agent to skip work it must always perform.

On demand: adapt the method

These skills add a method needed only for a particular request or evidence state. Their conditions keep the method available without imposing it on work where it does not apply.

The split keeps instructions relevant

If every core skill were always loaded, irrelevant procedures would compete for attention and could introduce work the request does not need. If foundational skills were on demand, required safeguards could disappear from an invocation.

The existing agent definitions below show how the same boundary is applied in practice.

Technology Extensions Setup Process

Project Configurator owns the setup process. It inspects the target repository, selects technology extensions from source evidence, records the intended configuration in PROJECT.yaml, renders the operational folder guidance, and verifies that the result matches the recorded intent.

Development precursor — outside setup

Actor: Methodology maintainer. Technology skills and their detection metadata are authored, validated, and shared before setup begins. Project Configurator consumes that existing collection; it does not create technology skills while configuring a target project.

Actor: Project Configurator
1. Inspect project scopes

Inventory the project family, application tiers, manifests, source roots, tests, and folder boundaries without combining unrelated sibling technologies.

Actor: Project Configurator
2. Match technology evidence

Run setup-time detection for representative folders. Accept a technology skill only when one complete evidence rule matches, and retain missing requirements, conflicts, and explicit no-variant results.

Actor: Project Configurator
3. Write the intermediate configuration

Create or update the root PROJECT.yaml before generating operational guidance. This reviewable intent log records the selected agent definitions, folder skillsets, source evidence, routing decisions, and planned AGENTS.md locations.

Actors: Maintainer and Project Configurator
4. Review or correct the intent

The maintainer can edit PROJECT.yaml to force a correction to the intended configuration. Project Configurator then reconciles that edit with repository evidence and bundle constraints before continuing.

Actor: Project Configurator
5. Render folder guidance

Generate root or nested AGENTS.md files with unconditional folder skillsets. For Claude Code, create a thin colocated CLAUDE.md bridge beside each applicable AGENTS.md.

Actor: Project Configurator
6. Verify the setup

Confirm that PROJECT.yaml, folder guidance, bridges, referenced agent definitions, skills, paths, and validation commands agree and that every required runtime capability is available.

Operational result — after setup

Actors: Coding, review, diagnosis, and verification agents. Ordinary work loads the established skillset for the active folder. These agents use the configuration output and do not rerun setup-time detection.

Setup-Time Detection Guardrails

Skill-owned activation

Each specialized skill owns its identifier and any-of activation branches. A branch can combine same-file path and extension checks, owning dependencies, manifest files, and parsed source imports.

Stable inputs only

Repository evidence and the selected folder scope are inputs. Agent names, prompt keywords, task wording, skill-read confirmations, and optional local commands are not.

Truthful results

The setup record includes the concrete matches, installed availability, missing required skills, exclusive conflicts, and explicit no-variant outcomes instead of inventing support.

One setup boundary

Detection runs when project configuration or technology boundaries change. Runtime activation by a particular vendor remains a separate verification concern.
