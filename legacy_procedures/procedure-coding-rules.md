# Procedure: Coding Rules

This document outlines the standard rules and best practices for writing code within this project, derived from the code-review-checklist.md and refactoring-plan-gemini.md. Adherence to these rules ensures consistency, readability, maintainability, and correctness.

## 0. File Header and Verification

- **RULE:** Every file (including source code, scripts, documentation) **MUST** include a standard header comment block (or equivalent metadata for non-comment supporting formats like JSON) containing the following information:

  - Copyright notice: "Copyright (c) 2025 Martin Bechard [martin.bechard@DevConsult.ca](mailto:martin.bechard@DevConsult.ca)"
  - License: "This software is licensed under the MIT License."
  - File path: Relative path from the project root (e.g., `src/services/MyService.ts`).
  - Credit: "This was generated by <AI Model Name> with the assistance of my human mentor"
  - 1-line summary: A brief description of the file's purpose.
  - Witty remark: A short, relevant, and lighthearted comment related to the file's content.
  - **BECAUSE:** Ensures consistent metadata, attribution, and licensing across all project files.
    - **BECAUSE:** Provides essential context about the file's origin, purpose, and legal status.

- **RULE:** Every `.ts` implementation file **MUST** start with a header comment referencing the primary design document that defines its behavior and interfaces. In addition to the regular header information, reference the design:
  - **FORMAT:** // Design: [[path/to/relevant-design-document.md]]
  - **NOTE:** The [[definitions.md]] file also links classes/interfaces/types defined in this file to their design document and canonical code path.
  - **BECAUSE:** This creates an explicit link between the implementation and its specification, making it easy to find the intended design.
    - **BECAUSE:** It aids maintainability by ensuring developers consult the correct design before making changes.
- **RULE:** Before modifying existing code or after adding new code, **ALWAYS** verify the changes against the design document referenced in the header and ensure alignment with the definition listed in [[definitions.md]]. If there is a discrepancy, raise the issue with the user.

  - **BECAUSE:** Ensures the implementation continues to align with the agreed-upon specification, preventing architectural drift.
    - **BECAUSE:** Catches unintended deviations from the design early in the development process.

- **RULE:** Any proposed major design changes or significant refactorings **MUST** be discussed and approved by the user/mentor before implementation.

  - **BECAUSE:** Ensures significant architectural shifts align with project goals and maintainability standards before effort is invested.
    - **BECAUSE:** Prevents potentially costly rework if major changes are later deemed undesirable or misaligned.

- **RULE:** The `design/modules/` directory contains the reference design documentation for most modules. When changing module code, the corresponding design document in `design/modules/` **MUST** be consulted and updated to reflect the changes. Conversely, if code seems incorrect or incomplete, the design document **SHOULD** be used as the primary reference to understand or reconstitute the intended logic.

  - **BECAUSE:** This ensures the design documentation remains the single source of truth for module behavior and facilitates consistency between design and implementation.
    - **BECAUSE:** Keeping design and code synchronized reduces confusion and prevents architectural drift.
    - **BECAUSE:** Design documents provide essential context for understanding complex logic or recovering intended behavior.

- **RULE:** After implementing changes and ensuring tests pass, **ALWAYS** perform a code review using code-review-checklist.md.

  - **BECAUSE:** Code reviews identify potential issues, enforce standards, and improve overall code quality beyond what automated tests can catch.
    - **BECAUSE:** It provides a human check for logic, readability, maintainability, and adherence to non-testable best practices.

- **RULE:** For custom JSON files (i.e., those not adhering to a standard external schema like `package.json` or `tsconfig.json`), metadata **MUST** be included using dummy properties prefixed with an underscore: `_copyright`, `_license`, `_path`.
  - **EXAMPLE:** `{ "_copyright": "Copyright (c) 2025...", "_license": "MIT", "_path": "config/custom-settings.json", "mySetting": true }`
  - **BECAUSE:** Provides a standardized way to embed required metadata directly within custom JSON structures where comments are not supported.
    - **BECAUSE:** Distinguishes project metadata from actual configuration data using the underscore prefix.

## 1. Naming Conventions

- **RULE:** File names containing a single primary class or interface **MUST** be named with the same PascalCase casing as the class/interface (e.g., `StatusManager.ts` for class `StatusManager`). The filename **MUST** match the `Design Code Path` specified in [[definitions.md]] if the class/interface is listed there. Note that files exceeding the line limit may be converted into module folders (see Section 2).
  - **BECAUSE:** This makes it easy to locate the definition of a class or interface using predictable conventions linked to the central registry.
    - **BECAUSE:** It creates a predictable mapping between the artifact name (class/interface) and the file system name, verifiable via definitions.md.
      - **BECAUSE:** Predictability reduces the cognitive load and time spent searching for files.
- **RULE:** File names for utility modules or collections of related functions **SHOULD** use camelCase or lowercase (e.g., `eventUtils.ts`, `errorHandler.ts`). File names for collections of related constants **SHOULD** use camelCase or lowercase and be placed in the `src/constants/` directory (e.g., `src/constants/apiEndpoints.ts`, `src/constants/workflowStates.ts`). Module entry points **MUST** be named `index.ts` and reside within their module folder. These files should also be listed in [[definitions.md]] if they define exported types/constants used elsewhere or serve as the canonical entry point for a module folder.
  - **BECAUSE:** This visually distinguishes utility modules, constant collections, and module entry points, and `definitions.md` provides their canonical path.
    - **BECAUSE:** This distinction helps developers quickly understand the likely contents and purpose of a file based solely on its name casing and registry entry.

- **RULE:** Use common naming conventions for JS/TS files based on their primary content.

  - **BECAUSE:** It improves code readability and maintainability by signaling the file's purpose.
    - **BECAUSE:** Consistent naming makes it easier to understand the type of entity defined within a file at a glance.
      - **BECAUSE:** Predictable naming reduces cognitive load and speeds up development.
  - **BECAUSE:** It aligns with established community practices, making the codebase more accessible.
    - **BECAUSE:** Following conventions makes the codebase more familiar to new developers.
  - **DETAILS:**
    - **PascalCase** for files containing:
      - Classes (unless file becomes a module folder)
      - React Components
      - Standalone Interfaces/Types defining a "thing" (e.g., models in `src/interfaces`, `src/types`) Example: `Agent.ts`, `UserProfile.ts`, `LoginButton.tsx`, `src/interfaces/Order.ts`
      - **BECAUSE:** PascalCase visually distinguishes constructors, components, and type definitions, aligning with the convention for the primary export.
    - **camelCase** or **lowercase** for files containing:
      - Utilities
      - Functions
      - Hooks (in React)
      - Generic helpers
      - Module entry points (`index.ts`) - *Strictly for re-exporting from files within the same module folder.*
      - Grouped type files (e.g., `workflowTypes.ts`) Example: `utils.ts`, `formatDate.ts`, `useAuth.ts`, `index.ts`, `src/types/configTypes.ts`
      - Collections of constants (e.g., `apiEndpoints.ts`, `stateVariableKeys.ts`) - *Typically placed in `src/constants/`.*
      - **BECAUSE:** camelCase/lowercase is standard for function-like entities, non-class/component modules, and collections. `index.ts` has a specific role as a module facade.
    - **kebab-case** for:
      - Configuration files
      - Package files
      - Test files (sometimes, team dependent) Example: `jest.config.ts`, `tsconfig.json`, `vite.config.ts`
      - **BECAUSE:** kebab-case is common for configuration and non-code assets, often preferred for OS compatibility and distinguishing them from source code modules.

- **RULE:** Variable, function, and method names **MUST** be clear, descriptive, and reflect their purpose using camelCase. Avoid cryptic abbreviations or overly generic names (e.g., data, manager, temp).

  - **BECAUSE:** Descriptive names improve code readability and make logic easier to understand without extensive comments.
    - **BECAUSE:** The name itself conveys the intent and nature of the data or action, reducing ambiguity.

- **RULE:** Function/method names **SHOULD** indicate their purpose through word choice:

  - Use **nouns** or noun phrases for functions that retrieve information without side effects (getters, accessors). Example: `orderCount`, `getUserProfile`, `isActive`.
  - Use **verbs** or verb phrases for functions that alter state or perform actions (setters, commands). Example: `updateStatus`, `calculateTotal`, `saveConfiguration`.
  - **BECAUSE:** This convention improves readability by clearly signaling the function's intent (query vs. command).
    - **BECAUSE:** It helps developers quickly understand whether calling a function might change the system's state.

- **RULE:** Avoid variable names that are easily confused (e.g., orderCount vs orderCounts if they represent different concepts). Be precise.
  - **BECAUSE:** Subtle differences in naming for distinct concepts can lead to hard-to-find bugs.
    - **BECAUSE:** Developers might accidentally use the wrong variable if their names are too similar, leading to incorrect calculations or logic.
- **RULE:** Naming conventions **MUST** be applied consistently across the entire codebase.

  - **BECAUSE:** Consistency reduces cognitive load and improves predictability.
    - **BECAUSE:** A uniform style makes the codebase feel cohesive and easier to read and navigate as a whole.

- **RULE:** Manifest constants (variables intended to never change at runtime) **MUST** be named using `ALL_CAPS_SNAKE_CASE`.
  - **EXAMPLE:** `const MAX_RETRIES = 5;`, `const DEFAULT_TIMEOUT_MS = 10000;`
  - **BECAUSE:** This is a widely accepted convention that clearly distinguishes constants from regular variables.
    - **BECAUSE:** It signals to readers that the value is fixed and should not be reassigned.

## 2. File Organization and Imports

- **RULE:** Files **MUST** be placed within standard top-level directories based on their category:
    - `src/`: All primary source code (TypeScript, JavaScript, etc.).
        - `src/interfaces/`: Shared interface definitions (`interface MyInterface {...}`). Each interface typically resides in its own file named `MyInterface.ts`.
        - `src/types/`: Shared type definitions (`type MyType = {...}`). Types can be grouped logically (e.g., `configTypes.ts`) or placed individually (`MyType.ts`).
        - `src/constants/`: Shared constant definitions (`const MY_CONST = ...`). Constants can be grouped logically into files (e.g., `apiEndpoints.ts`, `stateVariableKeys.ts`).
        - _Note:_ Specific module code (services, utilities, components) resides directly under `src` or within subdirectories representing logical modules (e.g., `src/services/`, `src/utils/`, `src/workflow/`).
    - `scripts/`: Utility scripts (Shell, Node.js, Python, etc.) used for build, deployment, or development tasks.
    - `design/`: Project documentation, design documents, diagrams, integration test plans, etc.
    - `test/`: Test code (unit, integration), test plans (unit), mocks, and fixtures.
    - **BECAUSE:** Establishes a conventional high-level project structure, making it easier to locate different types of artifacts and clearly separating shared contracts.
      - **BECAUSE:** Separates source code from supporting scripts, documentation, and tests, improving organization.

- **RULE:** Shared constants **MUST** be defined within files located in the `src/constants/` directory. Constants can be grouped logically into files named appropriately (e.g., `apiEndpoints.ts`, `stateVariableKeys.ts`). An `index.ts` file **MAY** be used within `src/constants/` solely to re-export constants from multiple files within that directory.
    - **BECAUSE:** Provides an explicit location for shared constants, aligning with the updated file naming rule and allowing logical grouping. Clarifies the acceptable use of `index.ts` within this specific directory.

- **RULE:** TypeScript interfaces used by multiple distinct modules (e.g., different services, or a service and a utility) **MUST** be defined in the `src/interfaces/` directory, typically one interface per file named `<InterfaceName>.ts`. Add the definition to `[[definitions.md]]`. Interfaces used *only* within a single module (or file) **MAY** remain within that module's scope (co-located or in a module-specific types file).
    - **EXAMPLE:** An `Order` interface used by `OrderService` and `ReportingService` belongs in `src/interfaces/Order.ts`. An internal helper interface used only within `OrderService` can stay in `OrderService.ts` or `src/services/OrderService/orderServiceTypes.ts`.
    - **BECAUSE:** Centralizes shared contracts (interfaces) for discoverability, consistency, and ease of import across module boundaries.
      - **BECAUSE:** Clearly separates public API contracts from internal implementation details.

- **RULE:** TypeScript types (type aliases) used by multiple distinct modules **MUST** be defined in the `src/types/` directory. They can be grouped logically (e.g., `src/types/apiTypes.ts`, `src/types/workflowTypes.ts`) or defined individually (e.g., `src/types/UserId.ts`). Add the definition to `[[definitions.md]]`. Types used *only* within a single module **MAY** remain within that module's scope.
    - **EXAMPLE:** A `WorkflowStatus` type used across several workflow components goes in `src/types/workflowTypes.ts`. A configuration type used only by `TradeExecutor` can remain private to it.
    - **BECAUSE:** Provides a central location for shared data structures (types) that are not necessarily interfaces, improving organization and reusability.
      - **BECAUSE:** Prevents scattering shared type definitions across various modules.

- **RULE:** Any single `.ts` file (like a class file `MyClass.ts`, or a utility file `myUtils.ts`) exceeding 200 lines of code **SHOULD** be refactored into a module folder.
    - Create a folder named after the original file (e.g., `MyClass/`, `myUtils/`).
    - Move the original file's contents into one or more files *inside* this new folder.
    - Create an `index.ts` file inside the folder that exports the public API of the module (the classes, functions, types, etc. that were previously exported by the single file).
    - Update `[[definitions.md]]` to reflect the new structure (the `Design Code Path` might now point to the `index.ts` or a specific file within the folder).
    - Update imports targeting the old file to target the new module folder (usually via its `index.ts`).
    - **EXAMPLE:** `src/services/ComplexProcessor.ts` (350 lines) becomes `src/services/ComplexProcessor/` containing `index.ts`, `processorLogic.ts`, `helperFunctions.ts`, etc. `index.ts` would export the necessary elements from the other files.
    - **BECAUSE:** Prevents excessively long files, which are hard to read, understand, and maintain.
      - **BECAUSE:** Promotes better organization of complex components by grouping related internal files.
      - **BECAUSE:** Enforces a clear public API for the component via the `index.ts` file.

- **RULE:** Module folders **MUST** contain an `index.ts` file. This file **MUST** only contain import/export statements to define the public interface of the module. It **MUST NOT** contain any implementation logic itself.
    - **EXAMPLE:** `src/MyModule/index.ts` might contain `export * from './MyClass'; export { helperFunction } from './helpers';`
    - **BECAUSE:** Provides a single, predictable entry point for importing from the module.
      - **BECAUSE:** Clearly defines the public contract of the module, hiding internal implementation details.

- **RULE:** Files **MUST** be organized in a logical directory structure under the standard top-level directories (`src`, `test`, etc.). Global system-wide modules or components **MAY** reside directly under `src` (e.g., `src/logging/`) or within functional groupings (e.g., `src/services/`, `src/utils/`). The canonical path for files defining key classes, interfaces, types, constants, or module entry points (`index.ts`) **MUST** be reflected in `[[definitions.md]]`.
    - **BECAUSE:** A clear structure, documented in `definitions.md`, makes code navigation easier and improves maintainability.

- **RULE:** Each exported class **SHOULD** reside in its own distinct file *within* its logical module (which might be a folder if the original class file grew too large or was designed as a module).
    - **BECAUSE:** Improves modularity and makes it easier to locate class definitions.

- **RULE:** Unit test files **MUST** be placed in a directory structure within the `test/` directory that mirrors the structure of the code module being tested in `src/`.
    - **EXAMPLE:** Tests for `src/services/workflow/BuyExecutor.ts` go in `test/services/workflow/BuyExecutor.test.ts`. Tests for a module `src/utils/formatting/` go in `test/utils/formatting/`. Tests for `src/constants/stateVariableKeys.ts` go in `test/constants/stateVariableKeys.test.ts`.
    - **BECAUSE:** Keeps test files separate from source files while maintaining a clear and predictable relationship between code and its tests.

- **RULE:** Integration test plans **MUST** be placed in `design/integration-test-plans/`. Integration test *code* **SHOULD** be placed in `test/integration/`.
    - **BECAUSE:** Integration tests span multiple components, so co-location with a single source module is inappropriate.

- **RULE:** For large test suites (especially unit tests for complex classes/modules or integration tests), **MUST** create a dedicated directory named after the test suite (e.g., `test/services/workflow/QuoteTradeWorkflow/`) and split the tests into separate files, typically one file per logical section or `describe` block from the original test plan (e.g., `initialization.test.ts`, `orderSubmission.test.ts`).
    - **BECAUSE:** Splitting large test suites improves maintainability, readability, and processing efficiency.

- **RULE:** When splitting a large test file, **DO NOT** split it all at once. **MUST** migrate tests incrementally using a TDD-like approach.
    - **PROCESS:**
        1.  Create a sub-plan (e.g., `original-test-name-split.plan.md`) linking to the original test plan. List all tests (or `describe` blocks) to migrate and the target files.
        2.  Create the new directory structure (e.g., `test/module/MyTestSuite/`).
        3.  For each test (or logical group/`describe` block):
            *   Create the target test file (e.g., `section1.test.ts`).
            *   Move the test code from the original file to the new file.
            *   Run the specific new test file (`pnpm run test path/to/new/section1.test.ts`) and ensure it passes.
            *   Run the tests for the entire new directory (`pnpm run test path/to/new/directory/`) and ensure they pass.
            *   Update the sub-plan status.
        4.  Once all tests are migrated and passing in their new files and the directory run passes, delete the original large test file.
        5.  Update the main status file.
    - **BECAUSE:** Incremental migration minimizes risk, ensures continuous test validity, and prevents large, hard-to-debug failures.
    - **BECAUSE:** Using a sub-plan provides clear tracking for the migration process.
    - **BECAUSE:** Running both individual files and the full directory ensures both isolation and integration work correctly.
    - **REMINDER:** Be mindful of cost; ensure each step passes before proceeding.

- **RULE (Before Creating Files/Definitions):** Before creating a new file for a class, interface, type, or constant collection:
    1.  Check `[[definitions.md]]` thoroughly to see if a suitable definition already exists.
    2.  If it exists, do not create a new definition; import it from its canonical `Design Code Path` listed in `definitions.md`.
    3.  If it doesn't exist:
        * Determine the correct location based on these rules (e.g., `src/interfaces/`, `src/types/`, `src/constants/`, within a specific module folder, etc.).
        * Create the file/definition at that location.
        * **Immediately add** the new definition (Class/Interface/Type/Constants), its design doc link, and its canonical `Design Code Path` (pointing to the new file, e.g., `src/interfaces/MyInterface.ts`, `src/constants/stateVariableKeys.ts` or `src/MyModule/MyClass.ts`) to `[[definitions.md]]`.
    - **BECAUSE:** Prevents duplication and ensures `definitions.md` remains the single source of truth for definition locations, reflecting the established structure.

- **RULE (Imports):** To import an existing Class, Interface, Type, Enum, or Constant:
    1.  Find the definition in `[[definitions.md]]`.
    2.  Use the `Design Code Path` listed there as the basis for the import statement. Note that for modules refactored into folders, this path might point to the `index.ts` file (e.g., `src/MyModule/index.ts`) or a specific file within the module if necessary. Imports from shared locations will point to `src/interfaces/...`, `src/types/...`, or `src/constants/...`.
    3.  Apply project alias rules (e.g., `@/` for `src/`, `@test/` for `test/`) or relative paths (`./`, `../`) as appropriate to construct the final import path from the current file to the canonical `Design Code Path`.
    4.  **DO NOT** include the `.js` or `.ts` extension. Omit `index` when importing from an `index.ts` file (e.g., `import { MyClass } from '@/services/MyModule';` which resolves to `src/services/MyModule/index.ts`).
    - **EXAMPLE:**
        * If `definitions.md` lists `Interface: Order - ... - Design Code Path: src/interfaces/Order.ts` and you are in `src/services/OrderProcessor.ts`, the import is `import { Order } from '@/interfaces/Order';`.
        * If `definitions.md` lists `Class: StatusManager - ... - Design Code Path: src/services/workflow/StatusManager/index.ts` and you are in `src/services/workflow/BuyExecutor.ts`, the import might be `import { StatusManager } from './StatusManager';`. (assuming `StatusManager` is now a folder).
        * If `definitions.md` lists `Type: ConfigOption - ... - Design Code Path: src/types/configTypes.ts` and you are in `src/main.ts`, the import is `import { ConfigOption } from '@/types/configTypes';`.
        * If `definitions.md` lists `Constants: API_KEYS - ... - Design Code Path: src/constants/apiKeys.ts` and you are in `src/services/ApiService.ts`, the import is `import { API_KEYS } from '@/constants/apiKeys';`.
    - **BECAUSE:** Ensures imports always point to the single, authoritative definition listed in the registry, respecting the project's file structure rules.

- **RULE:** Use project alias paths (e.g., `@/interfaces`, `@/services`, `@/types`, `@/constants`, `@test/mocks`) for imports *across* different top-level modules/directories (e.g., importing from `src/interfaces` into `src/services`, or from `src` into `test`). Use relative paths (`./`, `../`) for imports *within* the same logical module/directory (e.g., importing `./helpers` from `./MyClass` within the `src/MyModule/` folder). Base the target path on the `Design Code Path` from `[[definitions.md]]`.
    - **BECAUSE:** Provides a balance between stable cross-module imports (aliases) and clear intra-module imports (relative paths), while ensuring the target is always the canonical definition.

- **RULE:** Remove any unused imports.
    - **BECAUSE:** Unused imports clutter the code and can cause confusion or hide potential issues during refactoring.

- **RULE:** Organize imports logically (e.g., external libraries first, then project aliases `@/`, then relative `../`, then sibling `./`).
    - **BECAUSE:** Consistent organization improves readability and makes dependencies clearer.

- **RULE:** Actively avoid and resolve circular dependencies between modules. Consulting `[[definitions.md]]` and considering the separation of interfaces/types/constants (`src/interfaces`, `src/types`, `src/constants`) can help identify and break potential circular references during design and implementation.
    - **BECAUSE:** Circular dependencies can lead to runtime errors, make code harder to understand/test, and hinder modularity. Separating contracts can often break cycles.

- **RULE:** Before assuming an import path is wrong (when importing from a path derived from `[[definitions.md]]`), verify the path construction rules (relative vs. alias, omitting `/index`) and use `list_files` to confirm the target file exists at the canonical `Design Code Path`. If the path seems correct and the file exists but resolution fails, ask for help.
    - **BECAUSE:** This avoids unnecessary troubleshooting of configuration files and focuses on actual path/file issues first, using `definitions.md` as the reference.

## 3. Code Structure and Readability

- **RULE:** Methods **SHOULD NOT** exceed 30-40 lines of code. Extract complex logic into private helper methods (using TDD during refactoring).
    - **BECAUSE:** Shorter methods are easier to understand, test, and maintain. They tend to have a single responsibility.
- **RULE:** Classes **SHOULD NOT** exceed 300 lines of code. Files exceeding 200 lines **SHOULD** be refactored into module folders (see Section 2). Ensure the class definition aligns with the design doc linked in `[[definitions.md]]`.
    - **BECAUSE:** Large classes/files often violate SRP and are hard to manage. Refactoring into modules improves organization. Linking to the design via `definitions.md` helps enforce focus.
- **RULE:** Functions or constructors accepting more than two parameters **MUST** use a single parameter object type. Name the type `<FunctionName>Params` or `<ClassName>Options`. Define this options type. If the type is used across multiple modules, place it in `src/types/` and add it to `[[definitions.md]]`. Otherwise, it can be co-located or placed within the module's private types.
    - **BECAUSE:** Parameter objects improve readability and maintainability, especially when adding/removing parameters. Registering shared types ensures canonical definition.
- **RULE:** Prioritize code clarity and correctness over premature abstraction or overly complex solutions solely for the sake of DRY. Apply abstractions during the Refactor step of TDD when patterns are clear.
    - **BECAUSE:** Maintainability and understandability are key. Correctness comes first.
- **RULE:** Use object destructuring (`...`) where it improves clarity when creating parameter objects.
    - **BECAUSE:** Can reduce boilerplate, but should not sacrifice readability.
- **RULE:** When implementing designs or refactoring, **FOCUS** on minimal, necessary changes to achieve the goal. Avoid over-engineering or introducing unnecessary complexity.
    - **BECAUSE:** Simpler solutions are generally easier to understand, test, and maintain.
    - **BECAUSE:** Minimizing the scope of change reduces the risk of introducing unintended side effects.

## 4. Types and Primitives

- **RULE (Before Creating Types/Interfaces):** Before defining a new type or interface:
    1.  Check `[[definitions.md]]` thoroughly to see if a suitable definition already exists.
    2.  If it exists, **import and use it** from its canonical `Design Code Path` (which might be in `src/interfaces/`, `src/types/`, or a specific module).
    3.  If it doesn't exist:
        * Determine if it's shared across modules. If yes, place Interfaces in `src/interfaces/` and Types in `src/types/`. If no, keep it private within the module.
        * Define the new type/interface in the correct location.
        * **Immediately add** the new definition, its design doc link (usually the doc for the module defining or primarily using it), and its canonical `Design Code Path` to `[[definitions.md]]`.
    - **BECAUSE:** Prevents duplication, promotes reuse, enforces the new structure, and keeps `definitions.md` the authoritative registry.

- **RULE:** Use `interface` for defining contracts that classes can implement or objects must adhere to, especially when shared across modules (place in `src/interfaces/`). Use `type` for defining the shape of data objects, function signatures, unions, intersections, etc., especially when shared (place in `src/types/`). Use `class` when behavior (methods), internal state management, or inheritance is required.
    - **BECAUSE:** This provides clear guidance on choosing the appropriate TypeScript construct based on the intended use case and where to place shared definitions.
    - **BECAUSE:** Separating shared `interface` contracts from shared `type` data shapes improves clarity of intent.

- **RULE:** Avoid "Primitive Obsession." Use specific types, enums, or classes (defined centrally in `src/types/`, `src/interfaces/`, or module-internally, and referenced in `[[definitions.md]]`) instead of raw primitives (string, number, boolean) to represent domain concepts.
    - **BECAUSE:** Using domain-specific types (found via `definitions.md`) improves type safety, self-documentation, and centralizes validation logic. Allows leveraging the new shared type locations.
- **RULE:** Groups of primitive types that are always used together **SHOULD** be encapsulated within an interface or type (placed appropriately in `src/interfaces/`, `src/types/`, or locally, and registered in `[[definitions.md]]` if shared).
    - **BECAUSE:** Improves cohesion and makes the relationship explicit. Registering ensures discoverability and correct placement.
- **RULE:** Handle optional values (`undefined` or `null`) correctly as per previous rules (omit members, use `??` judiciously only for meaningful defaults or required targets).
    - **BECAUSE:** Incorrect handling masks errors or leads to unexpected behavior.
- **RULE:** Use type assertions (`as any`, `as unknown as Type`) sparingly and only when absolutely necessary, with justification. Prioritize fixing underlying type mismatches.
    - **BECAUSE:** Type assertions bypass type safety and can hide bugs.
- **RULE:** Ensure code uses the correct types/interfaces based on design documents. Verify the type/interface comes from the canonical path specified in `[[definitions.md]]` (which now includes `src/interfaces` and `src/types`).
    - **BECAUSE:** Adherence to type contracts defined in designs and registered in `definitions.md` is crucial for system correctness, respecting the new structure.

## 5. Coupling and Cohesion

- **RULE:** Classes **MUST** have a single, well-defined responsibility (SRP), aligned with their design document referenced in `[[definitions.md]]`. If a class grows too large or complex, consider refactoring it into a module folder (see Section 2).
    - **BECAUSE:** High cohesion makes classes/modules easier to understand, test, and maintain.
- **RULE:** Methods **MUST** perform a single logical operation.
    - **BECAUSE:** High cohesion within methods improves readability and testability.
- **RULE:** Minimize coupling between modules/classes. Depend on interfaces (defined in `src/interfaces/` and registered in `[[definitions.md]]`) rather than concrete implementations where possible, especially across module boundaries.
    - **BECAUSE:** Loose coupling makes the system more flexible and resilient to change. Using registered, centrally located interfaces ensures consistent contracts.
- **RULE:** Do not directly access private members of other classes. Interact through public methods defined in the class interface. Modules should interact via their `index.ts` exports.
    - **BECAUSE:** Violates encapsulation and creates tight coupling. Module folders explicitly define their public API via `index.ts`.
- **RULE:** Use interfaces (registered in `[[definitions.md]]`, located in `src/interfaces/`) to define contracts between components/modules.
    - **BECAUSE:** Interfaces promote loose coupling and allow easier substitution. `definitions.md` ensures everyone uses the same contract definition from the correct shared location.
- **RULE:** Avoid "Feature Envy." Move methods to the class/module whose data (defined in its registered interface/class/types) they operate on most.

    - **BECAUSE:** Improves cohesion and reduces coupling.

- **RULE:** Utility functions primarily used by a single class **SHOULD** be implemented as `static` methods within that class or as private functions within the class's file/module folder, unless they are needed independently or would significantly bloat the class file (potentially triggering a module folder refactor).
    - **BECAUSE:** This improves cohesion by keeping closely related logic together.
    - **BECAUSE:** It avoids polluting the global or module scope with utility functions that have a narrow scope of use.

## 6. Error Handling and Troubleshooting

- **RULE:** Follow the error handling strategy defined in [[/design/error-strategy.md]].
    - **BECAUSE:** Consistency makes the system more predictable and easier to debug.
- **RULE:** Avoid empty `catch` blocks. Log or re-throw appropriately.
    - **BECAUSE:** Empty catches swallow errors, hiding problems.
- **RULE:** Do not use exceptions for normal control flow. Use conditional logic.
    - **BECAUSE:** Misusing exceptions is inefficient and obscures logic.
- **RULE:** Provide clear, informative error messages, including context.
    - **BECAUSE:** Aids debugging and monitoring.
- **RULE:** Ensure adequate error logging/tracing using `Tracer`.
    - **BECAUSE:** Logs/traces are vital for diagnosing failures.
- **RULE:** When using `Tracer.log`, include variable names: `Tracer.log(\`Context: varA=${varA}, varB=${varB}\`);`
    - **BECAUSE:** Makes trace logs significantly easier to read and understand during debugging.
- **RULE (Troubleshooting):** When troubleshooting unexpected variable values, consider renaming variables for clarity if their current names are misleading. Update the code accordingly.
    - **BECAUSE:** Clear naming aids debugging by making intent explicit.

## 7. Comments

- **RULE:** Every exported Function, Interface, Type, Class, and global variable **MUST** be preceded by a JSDoc comment explaining its purpose.
    - For global variables, the comment **MUST** also explain the allowed range and meaning of values if applicable.
    - **BECAUSE:** Documentation alongside the code improves understanding and maintainability.
    - **BECAUSE:** Clear explanations make it easier for others (and future self) to use the code correctly.
- **RULE:** Write comments to explain _why_, not _what_ (code should explain _what_). Focus on rationale, business rules, workarounds.
    - **BECAUSE:** Provides context the code cannot convey.
- **RULE:** Remove commented-out code. Use version control.
    - **BECAUSE:** Commented code is noise and becomes outdated.
- **RULE:** Ensure comments are accurate and up-to-date. Fix or remove misleading ones.
    - **BECAUSE:** Incorrect comments are worse than no comments.
- **RULE:** Avoid excessive comments. Refactor for clarity instead (e.g., better naming, smaller functions, potentially refactoring large files into modules).
    - **BECAUSE:** Self-documenting code is preferred.
- **RULE:** Remove irrelevant development comments (// TODO, attributions). Use task trackers and commit messages.
    - **BECAUSE:** Such comments are noise and ineffective.

## 8. Conditional Logic

- **RULE:** Avoid large `switch` statements or long `if/else if/else` chains.
    - **BECAUSE:** Hard to read, maintain, extend; often violates OCP.
- **RULE:** Consider polymorphism (using interfaces from `src/interfaces/`) or maps/objects instead of complex conditionals.
    - **BECAUSE:** Often leads to cleaner, more extensible, and testable designs leveraging proper contracts.
