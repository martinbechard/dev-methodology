## Purpose

This document provides guidelines and heuristics for writing effective `TODO:` comments within design review documents (`<design-doc-name>-review.md`). Following these guidelines helps ensure feedback is clear, actionable, and consistent, facilitating efficient design refinement.

## General Review Approach (Before Writing TODOs)

**Prerequisites:**

1.  **Identify Documents:** Locate both the review document (`<ComponentName>-design-review.md`) and the corresponding original design document (`<ComponentName>-design.md`). Per convention, they should share the same base name and reside in the same folder.
2.  **Read Original Design First:** Thoroughly read the original design document (`<ComponentName>-design.md`) to understand the intended design and context _before_ evaluating the review document.

**Evaluation Steps:**

Once you have the context from the original design, evaluate the review document (`<ComponentName>-design-review.md`) as follows:

1.  **Read Through Findings:** Go through all the findings presented in the review document.
2.  **Evaluate Findings:** For each finding, critically evaluate if there are any inherent contradictions, vagueness, or potential errors in the finding itself, considering the context from the original design document.
3.  **Cross-Reference Design (Again):** If a finding still seems unclear or questionable after reading the original design, revisit specific sections of the original design document for detailed verification.
4.  **Note Critical Items:** Pay close attention to findings flagged as "Critical".
5.  **Consider Target Audience:** Assess the criticality and relevance of each finding based on the intended target user (e.g., a retail trader, not necessarily a professional trader). Does the issue significantly impact the retail user experience or functionality?

Once you have evaluated the initial findings with this approach, proceed to add specific, actionable `TODO:` comments using the heuristics below.

## Heuristics for TODO Comments

1.  **Be Specific:**

    - Pinpoint the exact part of the design needing attention (e.g., a specific function, section number, concept, variable name).
    - **Example (Good):** `TODO: Clarify the retry logic in `executeBuy` (Section 3.1).`
    - **Example (Bad):** `TODO: Logic needs work.`

2.  **Use Action Verbs:**

    - Start the TODO with a clear verb indicating the required action.
    - **Common Verbs:** `Update`, `Clarify`, `Define`, `Add`, `Remove`, `Verify`, `Propose`, `Fix`, `Ensure`, `Document`, `Investigate`.
    - **Example:** `TODO: Define the `BuyExecutorOptions` type and its properties.`

3.  **Provide Justification (Why?):**

    - Ensure the reason for the TODO is clear, either within the TODO itself or in the surrounding review text.
    - If the rationale isn't obvious, add a brief "BECAUSE..." or ask for justification.
    - **Example:** `TODO: Use an iterative retry loop instead of recursion BECAUSE recursion can lead to stack overflows.`
    - **Example:** `TODO: Expand on persistence needs - WHY is this needed here?`

4.  **Reference Standards/Docs:**

    - If the feedback relates to a project standard, guideline, or another document, reference it explicitly.
    - **Example:** `TODO: Update dependency injection approach to align with Object-Instantiation-Design-Directives.md.`

5.  **Distinguish Action Types:**

    - Use clear language to indicate if the TODO requires a direct fix, documentation update, a proposal, investigation, or deferral.
    - **Fix:** `TODO: Fix the calculation in `calculateLimitPrice`.`
    - **Doc:** `TODO: Document the state transition diagram for workflow activation.`
    - **Proposal:** `TODO: Propose a timeout mechanism using Promise.race for approval.`
    - **Investigate:** `TODO: Verify if there is responsibility overlap between LifecycleManager and StatusManager.`
    - **Defer:** `TODO: Add extensibility for custom persistence strategies to the future enhancements list.`

6.  **Define Scope (If Necessary):**

    - If the scope of the required action might be ambiguous, add a clarifying phrase.
    - **Example:** `TODO: Update error handling documentation *for this component*, referencing the general error strategy.`

7.  **Use Iterative Markers (TODO2, TODO3):**

    - For follow-up comments after a response has been provided to an initial TODO, use numbered markers (`TODO2:`, `TODO3:`) to maintain the conversation thread clearly.
    - **Example:**
      ```
      TODO: Clarify the meaning of "synchronous" event handling.
      Response: It means the caller waits...
      TODO2: OK, update the design document to explicitly state this definition.
      ```

8.  **Place TODOs Inline:**

    - Position each `TODO:` comment immediately following the specific finding, recommendation, or text block it relates to within the review document.
    - Avoid summarizing TODOs in a separate list at the end; keep them contextual.

9.  **Request Clarification on Findings:**
    - If a finding presented in the review document is unclear, seems contradictory, or cannot be verified even after checking the original design, add a `TODO:` asking for clarification _about the finding itself_. This signals that the review needs rework before the underlying design issue can be addressed.
    - **Example:** `TODO: This finding seems to contradict Section 4 of the design doc. Please clarify the discrepancy.`
    - **Example:** `TODO: Unclear what "inconsistent state" refers to here. Please provide specific examples or scenarios.`

## Example Application

```markdown
### 3. Potential Issues and Questions

- **Retry Logic:** The current recursive retry might cause stack overflows.
  `TODO: Fix this by using an iterative loop BECAUSE it's more robust for high retry counts.`

- **Timeout Handling:** The mechanism isn't defined.
  `TODO: Propose a specific timeout mechanism (e.g., using Promise.race) for approval and document it.`

- **`orderIdUnique`:** Source and uniqueness guarantee unclear.
  `TODO: Clarify in the documentation that `orderIdUnique` comes from the external TradeService and uniqueness is guaranteed externally.`

- **Finding on State:** The review mentions "inconsistent state" but doesn't provide details.
  `TODO: Unclear what "inconsistent state" refers to here. Please provide specific examples or scenarios.`
```

## Review Conclusion and Next Steps

After evaluating the initial review and adding your specific TODO comments, follow this process:

1.  **Decision Point:** Based on your evaluation and the nature of your TODOs, decide:

    - **Approve Review:** The review itself is satisfactory, and the TODOs primarily target changes needed in _other_ artifacts (e.g., the original design document, code implementation, test plans).
    - **Request Review Rework:** The review document itself needs changes (e.g., clarification on findings, addressing contradictions identified during your evaluation, incorporating responses to your TODOs). This is often triggered if you added TODOs asking for clarification _about the review findings_ (Heuristic #9).

2.  **If Review is Approved:**

    - Clearly indicate approval within the review document (e.g., adding an "Approved by [Your Name]" note).
    - Update the corresponding task in the project status document (e.g., `refactoring-plan-status.md`) to mark the review as approved.
    - Ensure the status document reflects the next steps derived from the TODOs (e.g., tasks to update the design document, implement code changes, etc.). The TODOs now serve as requirements for those subsequent tasks.

3.  **If Review Rework is Requested:**
    - Ensure your TODO comments clearly indicate the required changes _within the review document_. Use the iterative markers (`TODO:`, `TODO2:`, etc.) as needed.
    - Update the project status document (e.g., `refactoring-plan-status.md`) to indicate that the review requires rework (e.g., mark as "Rejected", "Rework Requested", or similar status).
    - The AI assistant will then provide an updated version of the review document, incorporating responses or changes based on your TODOs.
    - Repeat the "General Review Approach" and add further TODOs (`TODO2`, `TODO3`, etc.) to the updated review document until you are satisfied and can approve it.
    - Do **not** update the project status document to "Approved" until the review document itself is satisfactory and approved in step 2.
